# GlobalCloud智能体团队文档清理与补充完善建议

日期：2026-06-08  
状态：清理与补强建议 v1

## 1. 当前不建议做的清理

当前不建议立即执行：

1. 全量重命名文件
2. 大量移动文件到新目录
3. 删除历史参考稿
4. 删除模板或草案

原因：

1. 当前大量内部链接已经稳定
2. 当前阶段重点是首轮实施前验证，不是资料归档工程
3. 过早做大规模移动会制造新的引用风险和状态噪音

## 2. 当前建议执行的轻清理

### 2.1 以分类表作为使用边界

后续使用时统一遵守：

1. `README.md` 与总控入口只引用“主索引与总控入口 + 正式基线 + 当前阶段文档”
2. 模板、草案、历史参考不作为默认引用入口
3. 样本证据只能支撑当前阶段判断，不能反向提升总体完成状态

### 2.2 冻结当前正式基线层

建议把以下类别视为“当前正式基线层”：

1. 主索引与总控入口
2. 体系架构与治理基线
3. 团队治理与项目控制
4. 实施过程与仓库准入

### 2.3 把以下类别视为“阶段执行层”

1. 专项会话与预检
2. 首轮实施前验证包与执行记录
3. 运行前样本与执行骨架

### 2.4 把以下类别视为“历史参考层”

1. 历史参考与外部基础稿
2. 仅保留参考价值、不可反向覆盖当前口径的旧文档

## 4. 风险优先级建议

### P0

1. `Brain remote / gbrain import / rag export` 缺口
2. 链同运行前样本深度不足
3. 厂行运行前样本深度不足
4. 灵策 / 评衡 / 证验运行前样本深度不足
5. 数枢数据库拒绝 / DLQ / Replay live 样本不足

### P1

1. 模型治理未工单化
2. 界面与统一体验未全量责任化
3. 执行版包草案未覆盖全部专项

### P2

1. 文件命名一致性
2. 历史参考层归档
3. 模板与草案集中管理

## 5. 最终建议

当前正确路径不是“大清理”，而是：

1. 先按分类表和审计表稳定使用边界
2. 继续关闭正式实施开发前缺口
3. 等首轮实施前验证阶段收口后，再做目录级归档和命名整理
