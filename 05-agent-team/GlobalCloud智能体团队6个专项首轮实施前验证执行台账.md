# GlobalCloud智能体团队6个专项首轮实施前验证执行台账

日期：2026-06-08
状态：专项验证执行台账 v1
用途：把 6 个已进入“正式只读预检”阶段的专项首轮实施前验证包从“已定义”推进到“可逐项执行、可留证、可判定阻塞”的统一执行台账。
说明：仓图、接稳、评衡、证验 4 个会话当前为显性化启动前占位，未纳入此台账第一批执行段。

## 1. 使用口径

1. 本台账只用于“首轮实施前验证准备”阶段。
2. 本台账建立在 6 个专项正式只读预检结论和 6 个专项验证包已形成的前提上。
3. 本台账只收口运行前样本、验证入口和阻塞，不收口联调完成、试运行完成或生产完成。
4. 所有专项最高状态上限仍为 `partial`。

## 2. 统一执行字段

| 字段 | 含义 |
|---|---|
| 专项 | 当前验证专项 |
| 当前目标 | 本轮必须完成的验证目标 |
| 当前样本 | 已经具备或已定义的样本基础 |
| 待执行动作 | 下一步必须执行的只读或局部安全动作 |
| 待验证入口 | 进入验证的仓库入口、模块入口或运行入口 |
| 预计产出证据 | 本轮动作完成后应能拿到的证据 |
| 当前阻塞 | 本轮执行前已知阻塞 |
| 责任人 | 小即团队当前主责成员 |
| 状态上限 | 本轮动作最多允许达到的状态 |

## 4. 执行顺序

建议顺序固定为：

1. 宪衡 / WAES
2. 链同 / PVAOS + GPC
3. 厂行 / GFIS + Edge
4. 数枢 / AI 与数据底座
5. 灵策 / Brain（知源协同）
6. 灵策 / XiaoC（蚁后）+ Hermes/XGD（大象）+ XiaoG

原因：

1. 先治理，再业务，再执行，再数据，再知识，再 AI 审计。
2. 前 3 项优先解决“主链是否有验证入口”问题。
3. 后 3 项优先解决“运行前样本是否能闭环”问题。

## 5. 统一阻塞升级规则

出现以下情况，必须升级到小即总控：

1. 仓库入口与既有只读预检结论不一致
2. 专项边界与执行矩阵冲突
3. 样本要求必须进入可写或高风险动作才能确认
4. 同一专项连续两轮没有新增证据

## 6. 当前正式结论

当前下一步不再是补规范，而是按本台账逐专项执行首轮实施前验证动作，并把结果回写到：

1. 对应专项首轮实施前验证包
2. 总控周报
3. 小即控制塔
4. 后续的联调前入口判断
