## 2. 包级责任分解

### 2.1 WAES 治理与控制塔包

- 主责成员：宪衡
- 主要交付物：
  1. 治理对象
  2. 治理事件
  3. 状态机
  4. 门禁矩阵
  5. 证据规则
  6. AI 授权规则
- 不得越权：
  1. 不得把 WAES 写成事务审批中心
  2. 不得把业务事实写入 WAES

### 2.2 PVAOS 生态入口包

- 主责成员：链同
- 主要交付物：
  1. 组织模型
  2. 项目模型
  3. 伙伴入口模型
  4. 门户与身份上下文说明
- 不得越权：
  1. 不得将 PVAOS 写成工厂主账
  2. 不得承接生产执行逻辑

### 2.3 GPC 运营协同包

- 主责成员：链同
- 主要交付物：
  1. 平台订单协同
  2. ASN
  3. 预约
  4. 运输
  5. POD
  6. 外部异常
  7. 多厂协同
- 不得越权：
  1. 不得承接工厂主账
  2. 不得替代 WAES 做治理审批

### 2.4 GFIS 工厂执行包

- 主责成员：厂行
- 主要交付物：
  1. 工厂订单
  2. 工单
  3. 质量
  4. 库存
  5. 批次
  6. LES
  7. EAM
  8. 发货出库事实
- 不得越权：
  1. 不得把 GFIS 写成区域供应链平台
  2. 不得允许外部系统直写工厂主账

### 2.5 Edge 边缘接入包

- 主责成员：厂行
- 主要交付物：
  1. 采集链路
  2. 协议转换
  3. 缓存补传
  4. 去重回执
  5. OT/IT 边界
- 不得越权：
  1. 不得将 Edge 写成业务主账
  2. 不得把采集信号等同于业务确认

### 2.6 AI 与数据底座包

- 主责成员：数枢
- 主要交付物：
  1. 数据库边界
  2. 审计库
  3. 指标与时序库
  4. 读模型
  5. Replay / DLQ / 补偿
  6. 资源仓库映射
- 不得越权：
  1. 不得引入共享主账
  2. 不得允许跨系统直写

### 2.7 企业级知识主存与知识引擎包

- 主责成员：知源
- 主要交付物：
  1. 知识主存模型
  2. 知识版本与发布链
  3. 知识引擎边界
  4. 知识审计与失效治理
- 不得越权：
  1. 不得将知识引擎写成知识真源
  2. 不得允许 AI 自动改正式知识版本

### 2.8 XiaoC（蚁后）+ Hermes/XGD（大象）+ XiaoG AI 服务包

- 主责成员：灵策
- 主要交付物：
  1. Prompt 模板
  2. 模型配置
  3. Agent 分层
  4. 工具权限
  5. 多端交互机制
- 不得越权：
  1. 不得允许 AI 直接写业务主账
  2. 不得允许 AI 绕过 WAES 授权


### 2.9 MMC 管理配置中心包

- 主责成员：宪衡
- 主要交付物：
  1. 治理模板基线
  2. 配置标准化模型
  3. 管理控制台
  4. 项目初始化模板
  5. 配置版本与发布链
- 不得越权：
  1. 不得将 MMC 写成业务主账
  2. 不得允许 AI 自动改正式配置版本

### 2.10 Brain 智能知识平台包

- 主责成员：灵策
- 主要交付物：
  1. WiIKI 知识 UI 层
  2. KDS↔Brain 联邦同步机制
  3. 知识搜索与展示
  4. 用户知识协作闭环
  5. 跨项目知识入口
- 不得越权：
  1. 不得替代 KDS 作为知识真源
  2. 不得允许 UI 层直写知识主存

### 2.11 PKC 个人知识工作台包

- 主责成员：链同
- 主要交付物：
  1. 个人仪表盘
  2. 端到端用户知识闭环
  3. 个人知识库与收藏
  4. 任务与通知中心
  5. 跨项目状态同步接口
- 不得越权：
  1. 不得将个人知识库写成团队知识真源
  2. 不得允许 PKC 直写其他项目主账

### 2.12 XiaoG 本地语音助手包

- 主责成员：接稳
- 主要交付物：
  1. ESP32 语音硬件接入规范
  2. WebSocket 连接可靠性机制
  3. 本地语音交互协议
  4. 离线缓存与补传
  5. OT/IT 边界安全
- 不得越权：
  1. 不得将 XiaoG 写成云端 AI 服务
  2. 不得允许语音指令直写业务主账

## 3. 包级验收共性要求

每个交付包验收前必须具备：

1. 边界定义
2. 主要对象
3. 主要事件或接口
4. 数据边界
5. 验收场景
6. 风险与阻塞说明

缺一项，不得判定完成。

## 4. 模型使用规则

1. 各交付包必须优先遵守上表中的默认主模型和默认副模型。
2. `GPT-5 nano` 不得用于架构裁决、状态升级建议、验收判断和主线代码修改。
3. 若使用 `GPT-5 mini` 出现连续两次返工、边界冲突或结论不稳，必须升级到 `GPT-5.2-Codex`。
4. 模型升级必须由小即确认，并在控制塔与 PMBOK 台账中记录。
