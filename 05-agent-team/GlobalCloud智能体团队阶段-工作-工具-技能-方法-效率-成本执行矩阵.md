## 1. 使用原则

1. 先看阶段，再选工作方法。
2. 先看环境门，再选工具。
3. 先看证据类型，再判断状态上限。
4. 没有落在矩阵内的动作，不默认执行。
5. 所有动作同时受 Harness Engineering 和 PMBOK 约束。

## 3. 效率分级规则

### 3.1 高效率任务

适用：

1. 仓库预检
2. 台账同步
3. 落点抽样
4. 差距汇总

要求：

1. 优先 shell + `rg`
2. 优先专项线程并行
3. 主线程只做收口

### 3.2 中效率任务

适用：

1. 设计基线
2. 规范收口
3. 验证包设计

要求：

1. 正确性优先
2. 少返工
3. 强一致性

### 3.3 低效率但必要任务

适用：

1. 本地运行验证
2. Browser / Computer Use 验证
3. 联调样本采集
4. 试运行闭环确认

要求：

1. 不追求并发数量
2. 追求可复现与可留证

## 4. 成本控制规则

### 4.1 token 成本

1. 设计类：中高
2. 仓库预检类：低
3. 台账同步类：极低
4. 运行验证类：中等，但时间成本更重要

### 4.2 时间成本

1. 设计阶段以收口时间为主
2. 预检阶段以扫描时间为主
3. 运行验证阶段以环境与验证时间为主

### 4.3 返工成本

以下情况视为高返工成本，必须避免：

1. 多人反复改同一主文档
2. 未先定范围就启动多个专项线程
3. 用设计环境代替运行环境证明完成
4. 用仓库级证据代替联调证据

## 5. 当前阶段适用矩阵

截至当前，团队主要适用以下三类行：

1. 实施前准备 / 只读预检
2. 首轮实施前验证 / 运行前样本设计
3. 首轮实施前验证 / 台账同步

当前不适用：

1. 试运行环境的完成性判断
2. 生产环境的完成性判断
