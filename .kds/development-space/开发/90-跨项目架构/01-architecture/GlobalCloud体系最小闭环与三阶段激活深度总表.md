# GlobalCloud 体系最小闭环与三阶段激活深度总表

日期：2026-06-12
状态：v1.1 — 上调 XiaoG 阶段一激活深度；新增蚂蚁/蚁后/大象 Agent 执行体系
用途：以"最小闭环"替代"试点"概念，将全部 12 个项目纳入统一演进框架，按阶段定义激活深度，消除架构迭代断点。

---

## 0. 核心概念：从"试点"到"最小闭环"

### 0.1 问题

"试点"概念隐含一个风险——被标记为"试点"的项目先行，其余项目"以后再说"。这会在阶段切换时产生架构迭代断点：后期纳入的项目需要重新设计与已有系统的接口、重新对齐数据模型和治理规则，甚至发现早期设计未预留扩展点。

### 0.2 方案

**最小闭环不是"选几个项目先跑"，而是"全部 12 个项目在第一天就处于架构中，只是激活深度不同"。**

每个项目在阶段一就被定义了完整的架构角色、接口契约、数据边界和治理规则。阶段的推进只改变激活深度，不改变项目的架构存在。这意味着：

- 阶段一的 GPC 不是"不存在"，而是"已定义接口契约、数据模型和连接器合同，未启动运行时"
- 阶段二不是"加入 GPC"，而是"GPC 从薄激活升级为全激活"
- 任何时候都不会出现"突然多了一个系统不知道怎么接"的情况

### 0.3 激活深度定义

| 深度 | 含义 | 标志 |
|---|---|---|
| **全激活** | 运行时运行，持续产出/消费数据，参与 Loop Engineering 微循环 | 有 running evidence、有 loop-state、有 Harness 判定 |
| **活跃** | 运行时运行，接口在线，但范围受控（仅指定场景或指定数据域） | 有 evidence 样本、有部分 loop-state |
| **薄激活** | 架构定义完整（Manifest + 对象目录 + 接口契约 + 连接器合同 + 治理规则），运行时未启动 | 有 Manifest、有连接器合同、已纳入项目群交叉分析 |

### 1.2 蚂蚁/蚁后/大象：AI Agent 三层执行体系

XiaoC、XGD、XiaoG 不是三个平行的 AI 项目，而是一个**三层 Agent 执行体系**，按任务粒度和复杂度分工：

```
                        ┌──────────────────┐
                        │     大象          │
                        │     XGD           │
                        │  深度分析/复杂推理  │
                        │  全局调度/策略建议  │
                        │  "这个季度的产能   │
                        │   瓶颈在哪里？"    │
                        └────────┬─────────┘
                                 │ 分解为子任务
                        ┌────────┴─────────┐
                        │     蚁后          │
                        │  XiaoC / Hermes   │
                        │  编排/路由/分发    │
                        │  "查A线产能→查B线 │
                        │   订单→对比→汇总"  │
                        └────────┬─────────┘
                                 │ 分发到执行终端
              ┌──────────────────┼──────────────────┐
              │                  │                  │
         ┌────┴────┐       ┌────┴────┐       ┌────┴────┐
         │  蚂蚁    │       │  蚂蚁    │       │  蚂蚁    │
         │ XiaoG #1 │       │ XiaoG #2 │       │ XiaoG #3 │
         │ 查GFIS   │       │ 查GPC    │       │ 发通知   │
         │ 工单状态  │       │ 订单数据  │       │ 到PKC    │
         └─────────┘       └─────────┘       └─────────┘
              │                  │                  │
         ┌────┴────┐       ┌────┴────┐       ┌────┴────┐
         │  GFIS   │       │  GPC    │       │  PKC    │
         │ 工厂事实 │       │ 平台协同 │       │ 个人面板 │
         └─────────┘       └─────────┘       └─────────┘
```

**三层分工**：

| 层级 | 项目 | 比喻 | 职责 | 典型任务 |
|---|---|---|---|---|
| **重分析层** | XGD | 大象 | 深度推理、全局分析、复杂决策辅助 | 多维度产能瓶颈分析、跨工厂排程优化推演、根本原因分析 |
| **编排层** | XiaoC/Hermes | 蚁后 | 任务拆解、模型路由、Agent 调度、结果汇总 | 将"查全厂生产状况"拆成 3 个查询→分发给 3 个 XiaoG→汇总结果 |
| **执行层** | XiaoG | 蚂蚁 | **跨系统操作执行**。调用各业务系统 API、执行 AI 指令、回传结果 | 查 GFIS 工单、调 GPC 订单、推 PKC 通知、写 WAES 审计日志 |

**XiaoG 的关键地位**：XiaoG 是 AI 从"说"到"做"的唯一桥梁。没有 XiaoG，AI 只能出建议，人必须在 GFIS/PVAOS/GPC 里手动操作。有了 XiaoG，AI 建议可以直接转化为对业务系统的 API 调用（在 WAES 授权范围内）。XiaoG 的"蚂蚁"数量可以随场景扩展——工单查询蚂蚁、质量分析蚂蚁、排程建议蚂蚁、通知推送蚂蚁，每个蚂蚁专精一个业务系统的特定操作。

**这是 XiaoG 在阶段一就必须活跃的原因**：阶段一的 AI 虽然是"助手"（不写业务主账），但"查数据"本身就是跨系统操作——AI 需要 XiaoG 去 GFIS 查工单、去 KDS 查 SOP、去 PKC 发通知。没有执行层，AI 的"建议"就是空中楼阁。

**拓扑说明**：

- **GPCF** 位于顶部，以 0.0 节统一定义约束全部 12 个项目，自身全阶段全激活。
- **MMC、WAES、KDS** 是三条横向治理线：MMC 管策略配置，WAES 管治理授权，KDS 管知识真源。
- **Brain、PKC** 是两个人机交互面：Brain 面向知识消费，PKC 面向任务与通知。
- **XiaoC（蚁后）、XGD（大象）、XiaoG（蚂蚁）** 是 AI Agent 三层执行体系：XGD 做重分析，XiaoC 做编排路由，**XiaoG 做跨系统执行**。
- **GPC → GFIS → Edge** 是核心业务链路。
- **XiaoG 横向连接全部业务系统**：GFIS、GPC、PVAOS、PKC、WAES、KDS——它是 AI 指令落地的唯一执行通道。

### 2.2 XiaoG 在阶段一的具体执行范围

| 执行场景 | XiaoG 蚂蚁类型 | 调用的系统 | 说明 |
|---|---|---|---|
| NL 查询工单状态 | 工单查询蚂蚁 | GFIS REST API（只读） | 人通过 PKC 对话面板提问 → XiaoC 拆解 → XiaoG 调 GFIS API |
| 质量异常预警 | 质量监控蚂蚁 | GFIS REST API（只读）→ PKC 通知 API | XiaoG 定时查询 GFIS 质量数据，阈值触发时推送 PKC |
| SOP 检索 | 知识检索蚂蚁 | KDS → Brain 搜索 API | XiaoG 代用户查询 KDS 知识库，结果返回 Brain UI |
| AI 审计记录 | 审计写入蚂蚁 | WAES 证据 API（只写审计日志） | XiaoG 记录每次 AI 查询/建议的 trace，写入 WAES |
| 排程建议获取 | 排程查询蚂蚁 | GFIS 产能/工单 API（只读）→ PKC 看板 | XiaoG 拉取产能数据，经 XiaoC 分析后推至 PKC 看板 |

**关键**：XiaoG 在阶段一只做两类事——①只读查询（调各系统 GET API），②写审计日志（调 WAES 证据 API）。不写业务主账。这是"AI 助手"定位在执行层的体现。

### 3.1 AI 服务闭环中的 XiaoG 执行角色

AI 服务闭环中的关键链路：

```
用户意图（PKC 对话面板）
   → XiaoC（蚁后拆解任务）
     → XiaoG 蚂蚁#1：查 GFIS 工单状态
     → XiaoG 蚂蚁#2：查 KDS 相关 SOP
     → XiaoG 蚂蚁#3：查历史案例
   → XiaoC（蚁后汇总）
   → Brain（知识 UI 呈现）
   → 用户获得答案
```

XiaoG 在这个链路中承担的是"手"的角色——XiaoC（大脑）想清楚要查什么，XiaoG（手）去各个系统把数据拿回来。没有 XiaoG，XiaoC 只能在脑子里想，碰不到任何业务系统。

| XGD | 有仓库，无 Manifest | 薄激活：完成 Manifest + 深度分析接口规范 | P1 补 Manifest |
| **XiaoG** | 有仓库，无 Manifest | **活跃：蚂蚁执行层上线 —— 至少覆盖 GFIS 只读查询 + WAES 审计写入 + PKC 通知推送** | P1 补 Manifest |
| GPCF | 本工作区 | 全激活：持续运行 | Manifest 需升级 |

---

## 6. 总结

**GlobalCloud 的最小闭环 = 全部 12 个项目在第一天就定义了完整的架构角色、接口契约和治理规则。** 三阶段演进不引入新项目，不新增连接器编号，不改变系统主责和边界。

**AI Agent 三层执行体系是闭环运转的动力源**：
- **XGD（大象）** 做深度分析，提出"应该怎么做"
- **XiaoC（蚁后）** 做编排路由，决定"谁去做、怎么做"
- **XiaoG（蚂蚁）** 做跨系统执行，实际"动手做"——调 GFIS、查 GPC、推 PKC、写 WAES

XiaoG 在阶段一的活跃是 AI 从"旁观建议"到"实际参与系统交互"的质变点。没有 XiaoG 的执行层，AI 服务闭环就断了"手"——AI 只能出文字建议，无法触碰任何一个业务系统的 API。
