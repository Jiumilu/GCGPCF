# 数枢专项正式只读预检结论

日期：2026-06-08
专项：数枢
范围：AI 与数据底座中的结构化数据库域、读模型、指标、证据、Trace、Replay、DLQ、补偿机制
预检性质：跨仓库正式只读预检
当前状态：`partial`

## 1. 预检对象

数枢专项不是单一业务仓库，而是跨系统数据底座，因此本轮只读预检采用“文档基线 + 真实仓库入口”联合口径。

### 1.1 文档基线

- `GlobalCloud绿色供应链体系数据治理模型.md`
- `GlobalCloud绿色供应链体系系统-数据库边界总表.md`
- `GlobalCloud绿色供应链体系资源仓库-业务对象映射总表.md`
- `GlobalCloud绿色供应链体系事件合同.md`

### 1.2 真实仓库入口

- WAES：`/Users/lujunxiang/Documents/Codex Space/WAES`
- PVAOS：`/Users/lujunxiang/Documents/Codex Space/PVAOS`
- GFIS：`/Users/lujunxiang/Projects/GlobalCloud GFIS`
- GPC 原型：`/Users/lujunxiang/Projects/GlobalCloud GPC`

## 2. Git 与工作区事实

本轮跨仓库只读预检确认：

1. WAES、PVAOS、GFIS、GPC 当前都存在真实 Git 仓库入口。
2. GFIS 已确认存在 `forbidden_writes` 约束，支持外部直写阻断设计方向。
3. PVAOS、WAES、GFIS 当前均存在真实 dirty state，因此后续实施前验证必须避免覆盖既有改动。
4. GPC 当前仍为 Odoo 历史原型仓库，不应被误判为最终数据主线仓库。

## 3. 边界符合度

按文档基线与真实仓库联合抽样，数枢专项当前边界是自洽的：

1. 数据底座是跨系统能力，不存在单一“数枢主仓库”。
2. 当前口径明确禁止共享业务主账。
3. 当前口径明确禁止跨系统直写业务主账。
4. 当前口径明确要求跨系统协同通过 API、事件、证据引用和读模型完成。
5. 资源仓库只承担语义聚合，不承担业务主账。

## 4. 当前已确认的仓库级与文档级证据

已确认的关键证据包括：

1. `GlobalCloud绿色供应链体系数据治理模型.md`
2. `GlobalCloud绿色供应链体系系统-数据库边界总表.md`
3. `GlobalCloud绿色供应链体系资源仓库-业务对象映射总表.md`
4. `GlobalCloud绿色供应链体系事件合同.md`
5. `gcfis_custom/gcfis_custom/api.py` 中的 `forbidden_writes`

这些证据能证明：

1. 数枢域当前有清晰的数据边界设计。
2. “禁止共享主账、禁止跨系统直写”不是口号，至少在 GFIS 已出现实现侧约束迹象。
3. 当前已经具备进入数枢专项真实实施前验证的基础。

这些证据还不能证明：

1. 真实数据库权限策略已经生效
2. 读模型只读隔离已经生效
3. Event Bus / DLQ / Replay 已有运行前样本
4. Evidence Ledger / Trace 回指链路已被真实验证

## 5. 预检结论

### 5.1 已完成的判断

1. 数枢专项已完成第一轮跨仓库正式只读预检。
2. 当前数枢专项必须继续按“跨仓库数据底座”处理，而不是误建单一数枢业务仓库。
3. 当前数据库边界、读模型边界、资源仓库边界与事件合同边界没有发现明显内在冲突。

### 5.2 尚未完成的判断

1. 尚未完成真实数据库权限策略验证。
2. 尚未完成真实读模型只读隔离验证。
3. 尚未完成真实 Event Bus / DLQ / Replay / 补偿样本抽取。
4. 尚未完成真实 Evidence / Trace 回指样本抽取。

## 6. 当前阻塞

1. 当前没有统一的数据底座运行仓库入口。
2. 大部分关键证据仍停留在设计基线层，尚未进入运行前样本层。
3. 当前缺真实数据库、事件总线、重放与补偿服务的环境级样本。

## 7. 下一步允许动作

当前允许进入下一步：

1. 基于 WAES、PVAOS、GFIS、GPC 四仓继续抽取数据库边界与事件边界实现侧落点。
2. 准备跨系统直写阻断、读模型只读、DLQ/Replay、Evidence/Trace 回指的首轮实施前验证清单。
3. 把数枢专项推进到真实实施前验证层。

当前不允许升级为：

1. `ready_for_human_acceptance`
2. `complete`
3. 已联调完成
4. 已试运行完成
