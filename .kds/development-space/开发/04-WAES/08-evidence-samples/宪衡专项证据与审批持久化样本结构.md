# 宪衡专项证据与审批持久化样本结构

日期：2026-06-08
专项：宪衡
范围：WAES 治理与控制塔域
当前状态：`in_progress`

## 1. 目标

把 WAES 中 evidence / approval / audit 的持久化结构和回退机制收口成可直接引用的样本结构说明，作为后续抽取真实返回样本的基础。

## 2. 持久化仓储入口

已确认文件：

1. `src/app/audit/agentEvidenceLedgerRepository.ts`

已确认仓储接口：

1. `getWriteStatus()`
2. `load()`
3. `recordSession()`
4. `recordRun()`
5. `recordToolCall()`
6. `updateToolCall()`
7. `recordApprovalRequest()`
8. `recordEvidence()`
9. `searchEvidence()`
10. `clearLocal()`

当前判断：

- WAES 已实现 agent audit / evidence ledger 仓储抽象。
- evidence / approval 不只是领域对象，已经进入持久化接口层。

## 3. Ledger 状态结构

已确认核心状态对象：

1. `AgentEvidenceLedgerState`

已确认状态集合：

1. `sessions`
2. `messages`
3. `runs`
4. `toolCalls`
5. `approvalRequests`
6. `evidence`

已确认持久化来源状态：

1. `remote_api`
2. `remote_configured`
3. `local_storage`
4. `empty`

对应对象：

1. `AgentEvidenceLedgerPersistenceSource`
2. `AgentEvidenceLedgerStatus`

当前判断：

- WAES 已明确区分远端 API 真写入、远端已配置但未必写入、本地回退快照、空状态。
- 这和当前治理页、Workbench 首页上的“远端/本地回退”展示口径是一致的。

## 4. Approval 样本结构

已确认构造函数：

1. `createApprovalRequest(input, now)`

已确认关键字段：

1. `id`
2. `runId`
3. `title`
4. `reason`
5. `riskLevel`
6. `targetAction`
7. `impactScope`
8. `status`
9. `approverPolicy`
10. `evidenceIds`
11. `expiresAt`
12. `decidedBy`
13. `decidedAt`

当前判断：

- 审批请求不是一句状态，而是完整对象。
- 已包含风险级别、目标动作、影响范围、证据引用、审批策略、过期时间和决定人字段。

## 5. Evidence 样本结构

已确认构造函数：

1. `createEvidence(input, now)`

已确认关键字段：

1. `id`
2. `runId`
3. `type`
4. `title`
5. `source`
6. `summary`
7. `contextRefs`
8. `hash`
9. `relatedIds`
10. `inputHash`
11. `outputHash`
12. `failureReason`
13. `createdAt`
14. `visibilityPolicy`

已确认额外处理：

1. `redactAgentEvidenceText()` 会脱敏
2. `hashAgentEvidenceValue()` 会生成 `fnv1a-*` 哈希

当前判断：

- Evidence 已具备审计所需的最小闭环：对象关联、输入哈希、输出哈希、失败原因、可见性策略。
- 当前还缺真实 evidence 记录样本，不缺结构。

## 6. 远端写入与本地回退机制

已确认关键函数：

1. `persistWithRemote()`

已确认行为：

1. 先写本地快照
2. 若远端可用，则尝试 POST 到远端
3. 若远端失败，则保留本地回退快照

已确认写入状态返回：

1. `remote_api`
2. `local_storage`

对应消息语义：

1. `Agent Evidence Ledger write reached remote API; local fallback snapshot remains synced.`
2. `Agent Evidence Ledger remote API write failed; kept local fallback snapshot.`

当前判断：

- WAES 当前不是“只有远端才算成功”。
- 它已经内置了本地 fallback snapshot 机制，符合实施前验证阶段的回退口径。

## 7. 查询与汇总样本结构

已确认函数：

1. `searchEvidence(query)`

已确认查询维度：

1. `sessionId`
2. `runId`
3. `contextRefId`
4. `source`
5. `limit`

已确认返回结构：

1. `evidence`
2. `summary.totalCount`
3. `summary.latestEvidenceAt`
4. `summary.byType`

当前判断：

- WAES 已具备 evidence 检索与按类型聚合的基础结构。
- 当前还缺真实 search 返回样本，不缺检索 contract。

## 8. Workbench 展示侧证据

已确认文件：

1. `src/app/workbench/WorkbenchHome.tsx`

已确认展示口径：

1. `auditReadStatus.source === remote_api`
2. `auditWriteStatus.source === remote_api`
3. `local_storage`
4. `remote_configured`

当前判断：

- Workbench 首页已把 audit/evidence ledger 的远端读写状态显式展示出来。
- 后续只要拿到真实返回样本，就可以进一步验证 UI 是否正确反映写入来源与回退状态。

## 9. 当前已取得的样本层结论

当前已经拿到：

1. Approval 持久化结构
2. Evidence 持久化结构
3. Ledger 状态结构
4. 远端写入与本地回退机制
5. Evidence 查询与汇总结构
6. Workbench 展示侧状态入口

当前仍未拿到：

1. 真实 `approvalRequest` 记录样本
2. 真实 `evidence` 记录样本
3. 真实 `searchEvidence()` 返回样本
4. 真实远端写入成功样本
5. 真实远端失败后回退样本

## 10. 下一步

建议继续按这个顺序推进：

1. 从 `draftApprovalRuntime.ts` 抽审批阻断后写 evidence / approval 的链式样本
2. 从 `commandBus.ts` / `eventBus.ts` 抽 connector blocked / accepted / running 状态链
3. 如有必要，再进入页面级或测试级入口验证 UI 是否正确展示 ledger 状态
