# GlobalCloud项目群界面工程整体实施方案

日期：2026-06-22  
状态：项目群 UI 工程总控文件 / 执行版

## 1. 执行结论

本文件是 GlobalCloud 项目群界面工程总控文件。

它不再只作为执行前蓝图，而是作为项目群界面工程的统一总纲、实施主线、治理机制和专项索引入口。

本文件解决以下事项：

1. 项目群 UI 到底统一什么，允许差异什么。
2. 不同项目类型如何采用不同 UI 规范，而不是一把尺子压平所有项目。
3. `WAES -> Studio` 如何成为项目群专业工作台类的第一条正式实施主线。
4. UI 质量如何在 Loop 中实际执行，而不是停留在技能描述。
5. 例外、扩散、治理、评估如何形成长期可审计闭环。

本文件不声明任何项目因本方案自动进入 `accepted`、`integrated` 或 `production_ready`。

## 2. 总控定位

本方案在项目群中的定位是：

```text
项目群界面工程总控文件
=
总纲
+ 分类模型
+ 实施主线
+ 治理机制
+ 评估机制
+ 专项索引
```

本方案是项目群统一强制规范的总控入口。

强制边界如下：

1. 新增页面必须纳入本方案。
2. 高风险存量页面必须纳入本方案。
3. 低风险存量页面按阶段滚动纳入，不做首轮全量追溯。
4. `GPCF` 作为项目群控制塔，负责总控、归类审批、例外治理和状态纪律。

## 3. 当前项目群现实边界

当前项目群不是单一产品，而是多项目、多角色、多界面寄存器并存的工程体系。

真实边界如下：

1. `GPCF` 是总控治理仓和控制塔，不是业务主账。
2. `WAES` 不是纯后台，而是以项目为主要基础单元的治理执行主体；项目单元可以是工厂、供应链平台，或组合项目。
3. `GPC` 和 `GFIS` 都属于业务主线，但场景不同；前者偏协同主业务，后者偏高密度工厂执行。
4. `PVAOS`、`MMC` 更偏门户、组织、配置和治理支撑。
5. `Brain`、`KDS`、`PKC`、`Studio`、`XGD`、`XiaoC` 共同构成知识驱动和 AI 增强工作台群，但不能被粗暴当成一种页面。
6. `XiaoG` 更适合轻执行、移动或只读入口。
7. `WAS` 是语义契约与治理支撑源，不按普通业务前台产品对待。

因此，项目群 UI 总控必须采用分型治理，而不是单一风格治理。

## 4. 项目类型模型

### 4.1 一级分类模型

项目群 UI 一级分类固定为以下 7 类：

1. 控制塔类
2. 专业工作台类
3. 运营后台类
4. 分析决策类
5. 门户与展示类
6. 移动作业类
7. 系统配置与治理类

### 4.2 二级约束模型

一级分类之外，项目群统一采用以下二级约束：

1. 用户对象
2. 交互复杂度

执行规则：

1. 先按业务形态确定主类。
2. 再按用户对象和交互复杂度附加二级约束。
3. 每个项目必须有 `1` 个主类。
4. 每个项目允许有 `0-2` 个副类。
5. 设计规范、组件约束、验收门禁首先按主类执行。
6. 复合系统再由副类补充约束。

## 6. 统一与差异边界

### 6.1 项目群必须统一的对象

P0 统一对象如下：

1. 状态与标记类
2. 反馈与系统状态类
3. 审计与证据类
4. AI 交互类中的边界组件
5. 表格与数据呈现类中的核心子集
6. 按钮类中的语义层级
7. 表单编排类中的校验反馈

统一原则：

1. 先统一语义，不先统一装饰。
2. 先统一高风险组件，不先统一全部视觉细节。
3. 先统一证据、治理、AI 边界，再统一普通展示风格。

### 6.2 允许项目级差异保留的对象

以下对象允许项目级差异保留，但不得突破统一令牌、统一状态语义、统一证据边界和统一 AI 边界：

1. 门户与品牌表达类
2. 专业工作台布局类
3. 图表与分析呈现类
4. AI 交互外观类
5. 工程工具类

## 8. 分阶段实施方案

### Phase 0：总控执行版建立

目标：

1. 把总控文件从说明文档升级为执行版。
2. 把 UI gate 接入 Loop 强制执行。
3. 明确首条正式实施主线和专项索引。

完成标准：

1. 本文件成为执行版总控文件。
2. Loop 模板和 validator 保持通过。
3. 新增 UI 相关 round 必须按新口径执行。

### Phase 1：专业工作台母框架建立

目标：

1. 以 `WAES` 作为专业工作台类的优先基础框架。
2. 先评估差距，再形成升级方案。
3. 完成 `WAES -> Studio` 首条正式实施主线设计。

完成标准：

1. `WAES` 母框架差距评估形成受控文档。
2. `WAES` 母框架升级方案形成受控文档。
3. `Studio` 操作工作台验证方案形成受控文档。

### Phase 2：样板扩散与例外治理

目标：

1. 按 `Studio -> PKC -> XGD -> GPC -> GFIS` 进行专业工作台扩散。
2. 对不适配对象启用受控例外，而不是私自分叉。
3. 建立项目群周复盘与月评审节奏。

完成标准：

1. 至少形成一条真实扩散验证链。
2. 例外申请、审批、关闭全部进入 Loop 可追踪结构。
3. 样板和母框架不再停留在口头共识。

### Phase 3：基线候选与项目群扩散

目标：

1. 识别首批项目群基线态候选。
2. 建立第二阶段控制塔类和业务主线类候选。
3. 为后续旧文档整并建立现实前提。

完成标准：

1. `WAES` 专业工作台母框架进入首批候选池。
2. `Studio` 操作工作台验证页进入首批候选池。
3. 第二阶段候选池被明确登记。

## 9. 实施主线

### 9.1 第一条正式实施主线

项目群专业工作台类的第一条正式实施主线固定为：

`WAES -> Studio`

定义如下：

1. `WAES` 负责输出专业工作台母框架基线。
2. `Studio` 负责验证该基线在知识驱动协作、工程流程和 AI 增强复合工作台中的成立性。
3. 该主线不是试点，也不只是样板验证，而是第一条正式实施主线。

### 9.2 WAES 母框架的约束级别

`WAES` 被定义为专业工作台类的优先基础框架，而不是强制唯一框架。

执行规则：

1. 默认先复用 `WAES` 母框架。
2. 只有证明不适配时，才允许申请例外。
3. 不允许整套绕开母框架。

### 9.3 WAES 母框架的统一范围

`WAES` 母框架统一到以下三层：

1. 外壳层
2. 页面骨架层
3. 核心高风险组件层

但当前 `WAES` 不是最终标准件，而是待补差的基础母框架候选。

### 9.4 WAES 母框架补差优先级

补差并行推进，但主次固定为：

1. 第一优先：状态语义与证据语义
2. 第二优先：信息架构与页面骨架
3. 第三优先：核心组件与交互模式

## 12. 例外机制

### 12.1 例外审批强度

例外审批采用中等强度规则：

1. 项目必须先证明 `WAES` 母框架在哪些层不适配。
2. 例外只能针对局部，不允许整套绕开。
3. 例外不是自由分叉许可，而是受控豁免。

### 12.2 例外审批权

例外审批权由三方联合持有：

1. `GPCF`
2. `WAES`
3. 目标项目

### 12.3 例外申请最小内容

例外申请至少必须包含：

1. 不适配对象
2. 不适配原因
3. 影响范围
4. 替代方案
5. 不扩散承诺

### 12.4 例外在 Loop 中的记录字段

只要 round 涉及例外页面或例外组件，就必须显式写出以下字段：

1. `exception_id`
2. `exception_scope`
3. `exception_reason`
4. `exception_approved_by`
5. `exception_status_ceiling`

### 12.5 例外关闭条件

例外关闭必须同时满足以下 4 条：

1. 替代方案已落地
2. UI gate 已验证
3. 三方确认完成
4. 文档与 Loop 已回写收口

## 13. Loop 接入规则

### 13.1 何时必须触发 UI gate

满足以下任一条件必须触发 UI gate：

1. 改动产品界面或工作台
2. 改动 AI 对话页或 AI 侧栏
3. 改动 evidence/audit 界面
4. 用前端回归或浏览器可见性支持本轮结论
5. 声称“界面已可用”“界面已统一”“交互已闭环”
6. 本轮涉及例外页面或例外组件

### 13.2 何时不得越权

UI gate 或 UI 证据不得单独声明：

1. 业务完成
2. 客户验收完成
3. `accepted`
4. `integrated`
5. `production_ready`

### 13.3 历史债务策略

历史 round 已存在大量 UI 信号但没有显式 UI gate 结构时：

1. 不做批量补写
2. 从新 round 开始严格执行
3. 历史 round 作为可见性债务保留
4. 如需修复，只做 targeted annotation，不重写业务事实

## 14. 治理节奏

### 14.1 三层治理节奏

项目群 UI 治理节奏固定为：

1. 每轮：涉及 UI 的 Loop round 强制执行 UI gate
2. 每周：项目群 UI 治理复盘
3. 每月：项目群级 UI 架构评审

### 14.2 每周 UI 治理复盘最小检查项

1. 样板工程推进状态
2. 母框架补差状态
3. 例外清单变化
4. UI gate 执行质量
5. 高风险页面整改状态
6. 扩散风险

### 14.3 每月 UI 架构评审最小议程

1. 母框架是否升级
2. 样板工程是否通过
3. 一级分类与映射是否需要修订
4. 统一组件清单是否扩容
5. 例外是否被滥用
6. Loop 门禁是否需要升级

## 15. 评估机制

### 15.1 双轨评估

项目群 UI 统一采用双轨评估，但门禁优先：

1. 门禁决定能不能升级状态
2. 分数只决定成熟度和横向比较

### 15.2 关键门禁

以下 5 个为关键门禁：

1. `G1 Surface Structure`
2. `G4 Evidence And Governance`
3. `G5 AI Fact Separation`
4. `G8 Runtime Verification`
5. `G9 Scope Control`

只要关键门禁失败，就不得升级状态。

### 15.3 成熟度评分

成熟度评分采用 `G1-G9` 九门同轴成熟度评分：

1. 每门 `0-5`
2. 输出总体成熟度档位
3. 支持同类型项目横向比较
4. 关键门禁未过时，不得因高分升级状态

### 15.4 总体成熟度档位

项目群 UI 成熟度档位固定为：

1. 基础态
2. 可用态
3. 可复用态
4. 稳定态
5. 项目群基线态

### 15.5 项目群基线态准入条件

进入“项目群基线态”至少必须满足：

1. 关键门禁连续通过
2. 至少完成一个跨项目扩散验证
3. 例外受控且数量有限
4. 月度架构评审确认

## 16. WAES 母框架专项输出要求

### 16.1 第一控制成果

第一控制成果固定为：

《WAES 母框架差距评估》

### 16.2 差距评估固定维度

该评估至少覆盖以下 6 维：

1. 页面骨架差距
2. 状态语义差距
3. 证据与审计语义差距
4. AI 建议边界差距
5. 核心高风险组件差距
6. 可扩散性差距

### 16.3 差距评估输出结论

每个维度只允许以下四级结论：

1. 可直接复用
2. 需补强
3. 需重构
4. 不适配

### 16.4 可扩散性优先判断顺序

可扩散性判断优先序固定为：

`Studio -> PKC -> XGD -> GPC -> GFIS`

## 17. 文档整并规则

## 18. 界面工程标准工具链

项目群界面工程的标准工具链固定为以下 9 层：

1. `@product-design`：产品设计前置层，负责 `get-context -> ideate -> 选择方案`，用于整体界面、页面类、关键组件和结构性重设计的三方案产出。
2. `WAES` 现有界面框架：专业工作台类默认母框架，统一外壳层、页面骨架层和核心高风险组件层。
3. `ui-ux-pro-max`：设计系统、组件词汇表、令牌语义和页面类型约束来源。
4. `Figma`：视觉真值源、设计稿对照源和组件映射源。
5. `Storybook`：组件/页面可见性回归与变体登记源；仓内存在时必须优先复用。
6. `impeccable`：产品界面审计、硬化、润色和结构优化层。
7. `Playwright/browser`：真实运行验证、桌面/移动截图、键盘路径和可交互验证层。
8. `axe-core` / `Lighthouse`：无障碍、可读性和性能验证层。
9. `GPCF Loop UI Quality Gate`：最终门禁层，只输出 `ui_evidence_candidate`，不得越权声明业务完成。

执行规则：

1. 缺少视觉目标的整体界面、页面类、关键组件设计，不得直接进入实现，必须先经过 `@product-design`。
2. 专业工作台类默认先复用 `WAES` 母框架；只有证明不适配时才允许申请例外。
3. 工具不可用时必须显式登记 `Tools unavailable`，不得假装已执行。
4. 不允许只调用实现工具而绕过设计、验证和门禁层。

## 19. 工具上下文包与提示词能力体系

### 19.1 工具上下文包

所有 UI 设计、改造、审计和验证任务，都必须先形成统一 `工具上下文包`。

最小字段固定为：

1. 项目与业务角色
2. 页面类或组件类
3. 目标用户与关键任务
4. 现有视觉/代码/设计系统来源
5. `WAES` 复用层级
6. 约束、禁改项与非目标
7. 质量门禁与交付证据要求

缺少工具上下文包时：

1. 设计任务状态上限为 `ui_blocked`
2. 实现任务状态上限为 `partial`
3. 不得声称“设计已完成”“界面已统一”或“体验已闭环”

### 19.2 提示词能力体系

除上下文包外，项目群 UI 工程必须同时具备统一 `提示词能力`，且提示词目标固定对齐以下 5 类要求：

1. 功能准确：界面必须围绕真实功能与真实流程，不允许空想功能。
2. 界面美观：视觉层级、密度、节奏、状态表达和风格统一要成立。
3. 使用方便：关键动作路径短、可理解、可恢复、可发现。
4. 体验优秀：反馈及时、状态明确、阻断清楚、长文本与多端鲁棒。
5. 证据真实：输出必须带可核验的运行、截图、测试或审计证据。

项目群统一提示词 profile 固定为：

1. `functional-accuracy`
2. `visual-quality`
3. `usability-experience`
4. `governance-evidence`
5. `scope-control`

任何工具调用都不得只写“优化一下界面”这类模糊指令，至少要显式写出：

1. 目标对象
2. 成功标准
3. 禁止越权项
4. 输出格式
5. 验证方式

## 20. 三方案机制与执行闭环

### 20.1 三方案机制

以下对象必须至少形成 `3` 种界面设计方案：

1. 项目总界面
2. 高风险页面类
3. 关键组件类
4. 结构性重设计对象

执行规则：

1. 三方案由 `@product-design` 负责首轮产出。
2. 三方案必须共享统一体验骨架、统一令牌和统一风格样式，不允许 3 套完全不同体系。
3. 方案选择后，才能进入代码实现、页面接入和运行验证。
4. 小型缺陷修复、文案修正、轻微对齐调整可以复用既有选定方向，不强制重做三方案，但必须在 round 中写明 `Selected option = existing`。

### 20.2 执行闭环

项目群 UI 工程的标准执行闭环固定为：

```text
界面分类
-> 工具上下文包
-> @product-design 三方案
-> 选择方案
-> WAES/令牌/组件映射
-> 最小实现
-> 真实验证
-> UI gate 留证
```

Loop 中至少必须显式记录以下字段：

1. `Tool route`
2. `Context package`
3. `Prompt profile`
4. `Design options`
5. `Selected option`
6. `WAES baseline reuse`

### 20.3 能力演进

工具链不是静态名单，而是持续演进对象。

固定机制如下：

1. 每周复盘失败的工具上下文包和提示词。
2. 每月升级项目群 UI tool route、prompt profile 和 evidence 模板。
3. 任何被证明无效的提示词，不允许长期挂在规范里不清理。
4. 任何被证明有效的上下文包和提示词，必须回写到总控或专项规范。

## 21. 非声明边界

本文件不声明以下事项：

1. 项目群 UI 已全部统一
2. 所有项目均已完成样板页接入
3. 所有组件均已完成统一实现
4. 所有 UI 问题均已修复
5. 任何项目已因本方案自动进入 `accepted`
6. 任何项目已因本方案自动进入 `integrated`
7. 任何项目已因本方案自动进入 `production_ready`
