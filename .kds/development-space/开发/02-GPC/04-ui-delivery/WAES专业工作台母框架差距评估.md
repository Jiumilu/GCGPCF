# WAES专业工作台母框架差距评估

日期：2026-06-22  
状态：专业工作台母框架差距评估 v1  
定位：`WAES -> Studio` 首条正式实施主线的第一控制成果

## 1. 评估目的

本评估用于回答一个具体问题：

`WAES` 当前是否已经具备作为项目群专业工作台母框架的条件；如果没有，差在哪里，哪些部分可以直接复用，哪些必须补强或重做。

本评估服务于以下控制目标：

1. 为《WAES 专业工作台母框架升级方案》提供输入。
2. 为 `Studio` 操作工作台验证方案提供母框架基线。
3. 为后续 `PKC -> XGD -> GPC -> GFIS` 扩散路径提供例外和适配判断依据。

## 2. 评估边界

本评估只覆盖 `WAES` 作为专业工作台母框架候选的三个层次：

1. 外壳层
2. 页面骨架层
3. 核心高风险组件层

本评估不覆盖以下事项：

1. `WAES` 项目代码级逐组件盘点
2. `WAES` live runtime 视觉验证
3. 任一项目已经完成 UI 升级或扩散验证
4. `GFIS` 等高专业密度场景已经确认无例外可直接接入

也就是说，本评估是项目群总控口径下的受控差距评估，不等于 `WAES` 仓代码审计报告。

本评估基于以下受控依据形成：

当前引用事实摘要：

1. `WAES` 已被项目群总控确定为“以项目为主要基础单元的治理执行主体”，不是纯后台。
2. 统一体验骨架已把 `WAES 控制塔` 明确纳入“操作工作台”标准结构。
3. 既有差距清单已明确：`WAES` 具备治理与控制塔结构集中、证据与状态边界清晰的优势。
4. 既有差距清单同时明确：治理确认页模板、危险操作确认模式、AI 授权与治理提示区仍未形成统一母框架级收口。

## 4. 结论总览

总体结论：

`WAES` 可以被定义为项目群专业工作台的优先基础框架候选，但当前还不能直接被视为项目群专业工作台母框架标准件。

当前判断：

1. 它在证据、审计、治理状态表达上基础最好。
2. 它在工作台骨架方向上具备天然优势。
3. 它在“可直接扩散”为其他专业工作台方面仍缺统一封装和统一语义层。
4. 它更适合先成为“待补差的母框架候选”，再进入 `Studio` 验证，而不是被直接宣布为项目群基线。

## 5. 六维差距评估

### 5.1 结论矩阵

| 评估维度 | 当前结论 | 判断摘要 |
|---|---|---|
| 页面骨架差距 | `需补强` | 已有工作台方向，但未形成可复用的母框架级页面模板族 |
| 状态语义差距 | `需补强` | 治理/状态边界较清楚，但尚未上升为项目群专业工作台统一状态语义 |
| 证据与审计语义差距 | `可直接复用` | `WAES` 在证据、回指、审计、治理状态表达上基础最强 |
| AI 建议边界差距 | `需补强` | AI 授权、治理提示、采纳驳回区尚未形成统一工作台级实现 |
| 核心高风险组件差距 | `需补强` | 高风险组件已有规范方向，但尚未沉淀为 `WAES` 母框架组件组织法 |
| 可扩散性差距 | `需补强` | 对 `Studio / PKC / XGD / GPC` 可进入受控扩散评估，对 `GFIS` 暂不宜直接套用完整框架 |

### 5.2 页面骨架差距

结论：`需补强`

当前基础：

1. 统一体验骨架已定义“操作工作台”标准结构。
2. `WAES` 已被项目群认定为专业工作台类主类，同时具备控制塔副类和治理副类。
3. `WAES` 现有叙事天然适合承接“状态区 + 主任务区 + 风险区 + 证据区 + AI 辅助区”的工作台结构。

主要缺口：

1. 缺少主列表、详情、异常页、证据页与工作台页之间的统一骨架映射关系。
2. 缺少“项目单元视图”作为 `WAES` 页面骨架主对象的显式定义。
3. 缺少从 `WAES` 工作台骨架向 `Studio` 操作工作台骨架迁移时的保留层和可变层划分。

直接判断：

`WAES` 已有骨架方向，但还没有形成项目群可复制的页面模板族，因此不能直接进入“完整复用”。

### 5.3 状态语义差距

结论：`需补强`

当前基础：

1. 组件与令牌规范已经定义 `status-running`、`status-blocked`、`status-partial`、`status-review`、`status-complete`。
2. `WAES` 的天然职责就是治理、证据、状态门控和风险裁决。

主要缺口：

1. 业务状态、治理状态、风险状态、人工确认状态之间仍缺专业工作台统一语义层。
2. 面向“项目单元”的阶段状态、阻断状态、待补证状态、待人工确认状态没有形成统一字典。
3. `Studio`、`PKC`、`XGD` 等知识驱动工作台如何继承 `WAES` 状态语义，当前未定义。

直接判断：

`WAES` 具备状态治理优势，但还没有把这种优势收口成可复用的项目群状态语义标准。

### 5.4 证据与审计语义差距

结论：`可直接复用`

当前基础：

1. `WAES` 已被项目群长期定位为治理、证据、状态门控和审计平面。
2. 既有差距清单已明确其证据、状态、指标、治理确认逻辑边界相对清晰。
3. 总控方案已把控制塔类证据/审计页列入第二阶段基线候选。

可直接复用的对象：

1. 证据摘要区
2. 来源回指
3. 审计日志
4. 风险说明
5. 状态与确认关系

仍需注意：

1. “可直接复用”指语义层可复用，不代表组件实现层已经完成。
2. 证据卡、裁决面板、审计项、来源引用块仍需要进入升级方案中做统一组件化。

### 5.5 AI 建议边界差距

结论：`需补强`

当前基础：

1. 总控方案已经明确 AI 建议不得伪装成业务事实。
2. 统一体验骨架与对话模式规范已定义 AI 辅助区、引用区、建议区和禁止动作提示区。

主要缺口：

1. `WAES` 治理型 AI 与普通业务建议型 AI 还缺清晰分层。
2. AI 授权提示、治理说明、采纳/驳回、人工接管区没有形成统一工作台级模式。
3. 还缺少“治理确认型对话”在专业工作台主面中的落位规则。

直接判断：

`WAES` 若要成为母框架，必须先把 AI 建议边界做成项目群统一强约束，而不是继续停留在规则描述层。

### 5.6 核心高风险组件差距

结论：`需补强`

当前基础：

1. 项目群已经定义高风险组件类和 P0 必须统一对象。
2. `WAES` 与专业工作台母框架直接相关的高风险组件包括：状态类、审计与证据类、AI 边界组件、表格核心子集、反馈态组件、批量操作与上下文区。

主要缺口：

1. 缺少 `WAES` 母框架自己的核心组件组织法。
2. 缺少治理确认危险操作的统一确认模式。
3. 缺少“主表格 + 右侧证据/AI/审计上下文”的复合工作台组件编排标准。
4. 缺少可直接下沉到 `Studio` 的操作栏、引用区、建议区、裁决区组合模板。

直接判断：

`WAES` 的组件问题不是“没有组件类别”，而是“没有母框架级组织方式”。

### 5.7 可扩散性差距

结论：`需补强`

扩散优先序：

`Studio -> PKC -> XGD -> GPC -> GFIS`

当前判断：

| 目标项目 | 当前判断 | 说明 |
|---|---|---|
| Studio | `需补强` | 最适合作为第一验证场景，但需补知识驱动协作和 AI 增强区编排 |
| PKC | `需补强` | 与知识工作台亲缘度高，但信息密度和个人/团队入口需调整 |
| XGD | `需补强` | 可继承工作台骨架，但长任务和运行面需要更强分屏与状态反馈 |
| GPC | `需补强` | 可借骨架和状态/证据语义，但业务主线对象复杂度更高 |
| GFIS | `不适配` | 当前阶段不宜直接套用完整母框架，只适合局部借鉴和后续例外评估 |

直接判断：

`WAES` 当前更适合向知识驱动和治理相近的专业工作台扩散，不适合在未做例外评估前直接压到 `GFIS`。

## 6. 差距清单到改造方向的映射

| 差距项 | 改造方向 |
|---|---|
| 页面模板族不足 | 在升级方案中定义主列表、详情、异常、证据、工作台五类母模板 |
| 项目单元视图未定义 | 把“项目单元”确认为 `WAES` 工作台主对象 |
| 状态字典未统一 | 建立专业工作台统一状态语义层 |
| AI 治理提示未成体系 | 建立治理型 AI 建议区、引用区、采纳驳回区、人工接管区 |
| 高风险组件组织法缺失 | 建立 `WAES` 母框架核心组件编排规则 |
| 跨项目扩散缺边界 | 在升级方案中写死扩散顺序、例外口径和关闭条件 |

## 7. 评估结论

本轮结论是：

1. `WAES` 具备成为项目群专业工作台母框架的最佳候选基础。
2. 当前最强的是证据与审计语义，不是完整框架成熟度。
3. 当前最弱的是母框架化封装能力，而不是单点规范数量。
4. 进入下一步前，不应直接宣布 `WAES` 为项目群专业工作台基线。
5. 下一步应立即转入《WAES 专业工作台母框架升级方案》。

## 8. 下一步动作

1. 基于本评估产出《WAES 专业工作台母框架升级方案》。
2. 把页面模板族、状态语义、证据组件、AI 边界组件组织方式写成可实施条款。
3. 并行产出《Studio 操作工作台验证方案》，承接 `WAES -> Studio` 实施主线。
4. 后续真实 UI round 必须在 Loop 中记录本评估结论引用和例外状态。
