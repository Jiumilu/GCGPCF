# GlobalCloud 绿色供应链体系 P0 最小闭环界面验收矩阵

日期：2026-06-08
状态：P0 最小闭环界面验收矩阵 v1
用途：把 P0 最小闭环界面实施清单转成可验收矩阵，明确每个节点界面实施何时算通过、需要哪些证据、哪些不能误判为完成。

---

## 1. 验收原则

P0 最小闭环界面验收只检查四类问题：

1. 业务路径是否可达
2. 状态与证据是否可见
3. 风险操作与异常处理是否统一
4. AI 建议是否与业务事实分离

本矩阵不验“是否已经做成最终完美界面”，而验：

**是否已经达到 P0 闭环可用、界面不发散、后续可继续收口。**

---

## 2. 节点验收矩阵

| 节点 | 第一阶段是否允许新建界面 | 最低通过标准 | 必要证据 | 不得误判 |
|---|---|---|---|---|
| 项目初始化 | 否 | 可完成项目创建、基础配置、发布/验证入口承接 | 配置流程截图或结构说明、字段清单、入口说明 | 不得因“界面旧”就判不通过；不得因“有页面”就判最终统一完成 |
| 组织/伙伴接入 | 否 | 可查看、检索、录入、定位组织与伙伴 | 列表页/详情页结构说明、关键字段与操作说明 | 不得因未统一视觉就判失败；不得因已有页面就判最终交付完成 |
| 平台订单 | 是 | 列表、详情、状态、异常入口、证据区结构稳定；不再沿原型漂移 | 页面骨架、状态区、证据区、异常入口、组件接入说明 | 不得把 Odoo 原型扩展误判为正式主线通过 |
| 样品确认 / 打样签样 | 是 | 可从平台订单进入样品申请；可查看配方研发、样品打样、样品检测、客户签样、转量产门禁状态 | 样品状态区、签样文件入口、检测结果入口、转量产门禁证据 | 不得把平台订单状态或普通工厂订单状态误判为样品确认通过 |
| 工厂订单 | 否 | 可承接已签样或已完成转量产门禁的平台订单下传后的查看与处理 | 现有页面结构、状态与操作说明、转量产来源引用 | 不得因未重画界面就判失败；不得绕过样品确认门禁 |
| 工单 | 否 | 工单创建、查看、处理路径可达 | 现有工单页结构、主操作路径说明 | 不得因风格未统一就误判失败 |
| 发货 | 否 | 发货状态、回执、证据、异常入口可见 | 页面结构说明、发货状态区、回执/证据入口说明 | 不得因未独立重构就误判失败 |
| 签收 | 是 | 签收详情、回执、证据、异常结构稳定，签收确认路径统一 | 页面骨架、状态区、证据区、回执区、异常入口 | 不得继续沿历史原型自然生长后判为通过 |
| 异常 | 是 | 形成统一异常壳层，摘要、影响范围、责任人、关闭、复盘入口统一 | 异常页骨架、关闭逻辑、责任人区、复盘入口 | 不得允许不同系统继续各自长异常页 |
| WAES 状态/证据 | 否 | 状态、证据、治理确认分区明确 | 状态区、证据区、治理确认区结构说明 | 不得把治理确认页写成业务审批页 |
| 日报/复盘 | 否 | 可生成、查看、修订日报/复盘草案，AI 建议不冒充正式结论 | 报告区、AI 建议区、修订入口说明 | 不得把 AI 复盘草案写成正式治理结论 |

---

## 3. 通用通过条件

所有节点都必须同时满足：

1. 页面或入口可达
2. 当前对象状态可见
3. 证据或证据入口可见
4. 风险动作有统一确认方式
5. AI 建议与业务事实分离

如缺少上述任一项，不得判定该节点通过。

---

## 4. 通用禁止误判

以下情况一律不得写成“通过”：

1. 只有页面，没有状态与证据表达
2. 只有结构草图，没有操作路径
3. AI 建议和业务事实混在一起
4. 危险操作没有统一确认
5. 异常关闭没有统一逻辑
6. 治理确认被写成业务审批

---

## 5. 当前阶段结论

当前这份矩阵的意义是：

1. 为 P0 界面实施提供统一验收标准
2. 避免团队各自定义“差不多可用了”
3. 把“界面收口”从设计语言推进到验收语言

下一步应把高风险模块中最危险的一条链单独做透。
