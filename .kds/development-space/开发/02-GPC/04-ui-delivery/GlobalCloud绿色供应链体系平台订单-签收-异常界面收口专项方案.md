# GlobalCloud 绿色供应链体系平台订单-样品确认-签收-异常界面收口专项方案

日期：2026-06-08
状态：平台订单-样品确认-签收-异常界面收口专项方案 v1.1
用途：针对当前最危险的一条链，即平台订单 -> 样品确认 -> 签收 -> 异常，单独制定界面收口专项方案，确保最容易暴露拼凑感的链路优先收口。

---

## 1. 专项范围

本专项只覆盖四类页面：

1. 平台订单
2. 样品确认 / 打样签样
3. 签收
4. 异常

原因：

1. 直接影响 P0 最小闭环
2. 当前分级中风险最高
3. 最容易延续历史原型路径
4. 最容易让用户感知“不是一个产品”
5. 样品确认是平台订单转工厂量产的独立门禁，缺失后会造成未签样直接量产的风险

---

## 2. 专项目标

本专项不追求一次做完全部细节，而追求以下目标：

1. 平台订单形成正式主线页面骨架
2. 样品确认形成独立状态、证据和转量产门禁骨架
3. 签收形成正式主线页面骨架
4. 异常形成统一壳层
5. 四者统一状态区、证据区、异常入口、风险确认方式
6. 四者优先接入首批统一组件

---

## 3. 平台订单收口

### 3.1 结构目标

平台订单页必须统一为：

1. 列表页
2. 详情页
3. 状态区
4. 证据区
5. 异常入口

### 3.2 交互目标

统一：

1. 订单查询
2. 订单筛选
3. 订单详情查看
4. 订单异常进入
5. 风险操作确认

### 3.3 组件目标

必须接入：

1. 标准表格
2. 列表筛选条
3. 状态标签
4. 主按钮 / 次按钮 / 危险按钮
5. 证据卡片

### 3.4 风险约束

1. 不得继续沿 Odoo 原型自然扩展正式订单交互
2. 不得先做视觉统一、后补结构统一

---

## 4. 签收收口

### 4.1 结构目标

签收页必须统一为：

1. 签收摘要
2. 状态区
3. 回执区
4. 证据区
5. 异常入口

### 4.2 交互目标

统一：

1. 签收确认
2. 回执查看
3. 异常进入
4. 风险关闭确认

### 4.3 组件目标

必须接入：

1. 状态标签
2. 证据卡片
3. 审计时间线
4. 危险确认弹窗

### 4.4 风险约束

1. 不得把签收做成附属功能页
2. 不得让签收异常处理逻辑独立发散

---

## 5. 异常收口

### 5.1 结构目标

异常页必须统一为：

1. 异常摘要
2. 影响范围
3. 当前责任人
4. 处置动作
5. 证据区
6. 复盘入口

### 5.2 交互目标

统一：

1. 异常定位
2. 异常处置
3. 异常关闭
4. 复盘进入

### 5.3 组件目标

必须接入：

1. 状态标签
2. 证据卡片
3. 审计时间线
4. AI 建议卡片
5. 危险确认弹窗

### 5.4 风险约束

1. 不得让不同系统各自长异常页
2. 不得让异常关闭逻辑各自定义

---

## 6. 四节点统一约束

平台订单、样品确认、签收、异常四个节点必须统一：

1. 状态表达
2. 证据表达
3. 风险提示
4. 危险确认方式
5. 异常入口位置
6. AI 建议展示方式

这四个节点中任何一个不统一，用户都会直接感知拼凑感。样品确认还必须额外保证“未签样/未豁免不得转量产”的风险提示与证据入口。

---

## 7. 专项验收

本专项通过至少要求：

1. 平台订单、样品确认、签收、异常四类页面结构可并排比对
2. 样品确认页面具备独立状态、证据和转量产门禁
3. 状态区语义一致
4. 证据区结构一致
5. 异常入口和关闭逻辑一致
6. 风险操作确认一致
6. AI 建议和业务事实区分一致

未同时满足，不得宣称该专项完成。

---

## 8. 当前阶段结论

这份专项方案的意义是：

1. 把最危险的一条链从总规范里抽出来
2. 单独作为优先收口对象
3. 避免“全部都想做，结果最关键的一条链反而没做好”

下一步应继续形成：

1. 模块级样板页清单
2. 首批组件接入样板清单
