# WAES专业工作台母框架升级方案

日期：2026-06-22  
状态：专业工作台母框架升级方案 v1  
定位：`WAES -> Studio` 首条正式实施主线的第二控制成果

## 1. 方案目的

本方案用于把 [WAES专业工作台母框架差距评估](/Users/lujunxiang/Projects/GlobalCloud%20V0.0.1/GlobalCoud%20GPCF/04-ui-delivery/WAES专业工作台母框架差距评估.md) 的结论，转成可执行升级路径。

本方案只回答以下问题：

1. `WAES` 要升级成什么样的母框架。
2. 差距项如何映射为具体改造项。
3. 哪些先做，哪些后做。
4. 如何扩散到 `Studio -> PKC -> XGD -> GPC -> GFIS`。
5. 何时允许例外，何时必须收口。
6. Loop 中要如何验证，而不是只写成计划。

## 2. 目标母框架定义

### 2.1 母框架定位

升级后的 `WAES` 母框架，不是“治理后台模板”，而是项目群专业工作台类的优先基础框架。

它必须同时满足：

1. 能承接项目单元视图。
2. 能承接状态、风险、证据、审计、AI 边界五类高风险表达。
3. 能被 `Studio` 这类知识驱动协作工作台复用。
4. 能被 `PKC / XGD / GPC` 分层继承。
5. 对 `GFIS` 保持局部借鉴和例外评估，而不是强行完整套用。

### 2.2 母框架三层目标

#### 外壳层目标

1. 统一顶部、侧边、页面容器、项目切换和上下文入口。
2. 统一高风险工作台的主导航和辅助导航层级。
3. 统一“项目单元”进入方式和页面范围说明。

#### 页面骨架层目标

必须形成以下 5 类母模板：

1. 主列表模板
2. 详情模板
3. 异常处理模板
4. 证据/审计模板
5. 操作工作台模板

每类模板必须显式定义：

1. 主区
2. 状态区
3. 风险区
4. 证据区
5. AI 辅助区
6. 审计区

#### 核心高风险组件层目标

必须形成以下组件组织法：

1. 状态语义组件组
2. 证据与审计组件组
3. AI 边界组件组
4. 主表格与筛选操作组件组
5. 危险操作确认组件组
6. 分屏与上下文工作区组件组

## 3. 差距到改造项映射

| 差距项 | 升级改造项 |
|---|---|
| 页面模板族不足 | 建立 5 类母模板定义和模板间映射 |
| 项目单元视图缺失 | 把“项目单元”定义为 `WAES` 工作台主对象 |
| 状态字典未统一 | 建立专业工作台统一状态语义层 |
| 证据组件未组件化 | 统一证据卡、引用块、审计项、裁决面板 |
| AI 治理提示未成体系 | 统一治理型 AI 建议区、引用区、采纳/驳回区、人工接管区 |
| 高风险组件组织法缺失 | 建立主表格、右侧上下文、分屏工作区和危险操作确认模式 |
| 扩散缺边界 | 明确扩散顺序、例外口径、关闭条件和 Loop 验证要求 |

## 4. P0 / P1 / P2 优先级

### P0：必须先补

1. 专业工作台统一状态语义层
2. 证据与审计组件组
3. 治理型 AI 边界组件组
4. 操作工作台模板
5. 危险操作统一确认模式

P0 通过标准：

1. 状态、证据、AI、危险确认四条高风险语义不再分裂。
2. `WAES` 操作工作台模板可以稳定承接 `Studio` 验证。

### P1：模板族补齐

1. 主列表模板
2. 详情模板
3. 异常处理模板
4. 证据/审计模板
5. 主表格与筛选操作组件组
6. 分屏与上下文工作区组件组

P1 通过标准：

1. `Studio` 可以复用 `WAES` 母框架进入操作工作台验证。
2. `PKC` 和 `XGD` 可进入受控继承评估。

### P2：跨项目扩散收口

1. `PKC` 继承方案
2. `XGD` 继承方案
3. `GPC` 继承方案
4. `GFIS` 局部借鉴与例外边界

P2 通过标准：

1. 至少一个跨项目验证通过。
2. 例外机制真实运行。
3. 月度架构评审确认扩散结论。

## 5. 例外与不适配规则

### 5.1 默认规则

1. 专业工作台项目默认先复用 `WAES` 母框架。
2. 不允许项目整套绕开母框架另起一套。
3. 只允许针对局部骨架、局部组件、局部交互申请例外。

### 5.2 例外适用场景

当前明确可能触发例外的场景：

1. `GFIS` 高密度连续作业工作台
2. 长任务运行面超高信息密度场景
3. 移动轻执行场景
4. 特殊图表/工业信号呈现场景

### 5.3 例外申请约束

例外申请至少必须回答：

1. 哪一层不适配
2. 为什么不适配
3. 影响哪些页面类和组件类
4. 替代方案是什么
5. 是否承诺不向项目群标准反向扩散

## 6. 扩散路径

扩散顺序固定为：

`Studio -> PKC -> XGD -> GPC -> GFIS`

### 6.1 Studio

用途：

1. 验证知识驱动协作工作台是否能复用 `WAES` 母框架。
2. 验证 AI 边界组件和分屏工作区是否成立。

### 6.2 PKC

用途：

1. 验证个人/团队知识工作台是否可继承操作骨架和状态语义。
2. 验证较轻工作台在不丢失统一性的前提下如何保持高频可读性。

### 6.3 XGD

用途：

1. 验证长任务、重分析和运行辅助工作台如何继承母框架。
2. 补强状态反馈和分屏编排能力。

### 6.4 GPC

用途：

1. 验证协同主业务工作台如何继承骨架与证据语义。
2. 控制业务主线不再自由生长出第二套工作台。

### 6.5 GFIS

用途：

1. 只验证哪些语义层和局部组件可借鉴。
2. 当前阶段不以“完整框架复用”为目标。

## 7. Loop 验证要求

### 7.1 升级方案相关 round 必须写明

1. `UI scope = true`
2. `UI gate status`
3. 当前引用的母框架文档版本
4. 是否存在例外编号
5. 本轮 `status ceiling`

### 7.2 升级方案阶段性通过条件

#### P0 通过条件

1. `G1 Surface Structure` 不得失败
2. `G4 Evidence And Governance` 不得失败
3. `G5 AI Fact Separation` 不得失败
4. `G8 Runtime Verification` 至少给出真实验证说明
5. `G9 Scope Control` 不得失败

#### P1 通过条件

1. `Studio` 至少形成一个真实 UI gate round
2. 相关模板和组件在第二个页面场景可复用
3. 无新增未审批例外

#### P2 通过条件

1. 至少完成一个跨项目扩散验证
2. 例外被真实记录和关闭
3. 月度评审确认可以进入下一扩散阶段

## 8. 实施任务包

### 8.1 第一包：状态与证据语义包

交付：

1. 专业工作台状态语义层
2. 证据卡、来源回指、审计项、裁决面板结构
3. 危险操作确认规则

### 8.2 第二包：AI 边界收口包

交付：

1. 治理型 AI 建议区
2. 引用区
3. 采纳/驳回区
4. 人工接管区
5. 禁止动作提示区

### 8.3 第三包：模板族收口包

交付：

1. 主列表模板
2. 详情模板
3. 异常模板
4. 证据模板
5. 操作工作台模板

### 8.4 第四包：Studio 验证接入包

交付：

1. `Studio` 操作工作台验证方案
2. `WAES -> Studio` 适配保留层和可变层说明
3. `Studio` 首个真实 UI gate 验证轮次

## 9. 非声明边界

本方案不声明：

1. `WAES` 当前已经是项目群基线
2. `Studio` 已经完成继承验证
3. `PKC / XGD / GPC / GFIS` 已经进入扩散实施
4. 任一项目已经因本方案进入 `accepted`
5. 任一项目已经因本方案进入 `integrated`
