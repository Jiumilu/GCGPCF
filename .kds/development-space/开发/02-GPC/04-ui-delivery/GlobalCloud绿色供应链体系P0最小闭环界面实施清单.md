# GlobalCloud 绿色供应链体系 P0 最小闭环界面实施清单

日期：2026-06-08
状态：P0 最小闭环界面实施清单 v1
用途：把模块实施分级判定、界面分阶段治理规则、统一体验骨架规范、统一组件库建设计划、高风险模块收口计划，真正绑定到 P0 最小闭环实施主线中。

---

## 1. P0 最小闭环范围

本清单只覆盖以下 P0 闭环节点：

1. 项目初始化
2. 组织/伙伴接入
3. 平台订单
4. 样品确认 / 打样签样
5. 工厂订单
6. 工单
7. 发货
8. 签收
9. 异常
10. WAES 状态/证据
11. 日报/复盘

目标不是一次把所有界面做完，而是确保：

1. P0 闭环能跑通
2. 第一阶段不做无必要新界面
3. 必须新建或重构的界面从一开始就进入统一骨架和统一组件体系

---

## 2. 节点级实施策略

### 2.1 项目初始化

- 当前分级：产品内实现
- 第一阶段策略：**不新建独立新界面**
- 实施方式：优先在现有 PVAOS / WAES 配置能力内完成
- 界面要求：
  - 复用现有配置页
  - 逐步对齐统一配置页骨架
- 验收要求：
  - 可完成项目创建
  - 可完成基础配置
  - 可完成发布/验证入口承接

### 2.2 组织/伙伴接入

- 当前分级：直接可用
- 第一阶段策略：**不新建新界面**
- 实施方式：复用现有组织/伙伴列表与详情能力
- 界面要求：
  - 先不做视觉重构
  - 后续只做统一详情骨架校正
- 验收要求：
  - 可查看、可录入、可检索、可定位组织与伙伴

### 2.3 平台订单

- 当前分级：全面重构
- 第一阶段策略：**允许建设正式主线界面**
- 实施方式：按 `GPC` 正式主线建立
- 界面要求：
  - 必须进入统一列表页、详情页、异常入口、证据区骨架
  - 必须接入首批统一组件
- 验收要求：
  - 订单列表、详情、状态、证据、异常入口结构统一
  - 不再沿历史原型自然扩展

### 2.4 样品确认 / 打样签样

- 当前分级：二次开发
- 第一阶段策略：**作为平台订单与工厂订单之间的独立门禁**
- 实施方式：GPC 建立样品申请、客户签样、转量产入口；GFIS 承接配方研发、样品工单、样品检测；WAES 承接签样证据和转量产门禁
- 界面要求：
  - 平台订单详情中必须可见样品状态
  - 样品确认必须有独立状态区：待打样、打样中、待客户确认、已签样、驳回、已转量产
  - 必须区分样品确认文件、客户签样文件和量产订单文件
  - 未签样或未豁免时，量产动作必须有阻断提示
- 验收要求：
  - 可从平台订单进入样品申请
  - 可查看配方研发、样品打样、样品检测、客户签样状态
  - 可形成转量产门禁证据
  - 不得把普通工厂订单页面冒充样品确认闭环

### 2.5 工厂订单

- 当前分级：直接可用
- 第一阶段策略：**不新建新界面**
- 实施方式：复用 GFIS 现有能力
- 界面要求：
  - 仅做必要结构对齐
  - 不为统一外观而先重画
- 验收要求：
  - 能承接已签样或已完成转量产门禁的平台订单下传后的工厂订单查看与处理

### 2.6 工单

- 当前分级：直接可用
- 第一阶段策略：**不新建新界面**
- 实施方式：复用 GFIS 工单界面
- 界面要求：
  - 先保持原功能可达
  - 后续对齐统一工作台和详情骨架
- 验收要求：
  - 工单创建、查看、处理路径不被破坏

### 2.7 发货

- 当前分级：产品内实现
- 第一阶段策略：**尽量不新建新界面**
- 实施方式：优先在现有能力内配置与脚本承接
- 界面要求：
  - 后续进入统一详情结构与证据区
- 验收要求：
  - 发货状态可见
  - 回执、证据、异常入口可见

### 2.8 签收

- 当前分级：全面重构
- 第一阶段策略：**允许建设正式主线界面**
- 实施方式：按 `GPC` 正式主线建立
- 界面要求：
  - 必须统一签收详情、回执、证据、异常处理结构
  - 必须接入首批统一组件
- 验收要求：
  - 签收确认与签收异常有统一结构和统一风险提示

### 2.9 异常

- 当前分级：局部开发
- 第一阶段策略：**允许建立统一异常壳层**
- 实施方式：建立统一异常处理页或统一异常壳层
- 界面要求：
  - 统一异常摘要
  - 统一影响范围
  - 统一责任人
  - 统一关闭和复盘入口
- 验收要求：
  - P0 闭环中的异常不再各自为政

### 2.10 WAES 状态 / 证据

- 当前分级：治理面保留，界面以现有能力为主
- 第一阶段策略：**不为统一视觉单独新建治理页**
- 实施方式：复用 WAES 现有治理/证据承接面，逐步按统一骨架校正
- 界面要求：
  - 状态、证据、治理确认分区明确
  - 与业务确认界面保持边界
- 验收要求：
  - 可查看状态
  - 可查看证据
  - 可完成治理确认

### 2.11 日报 / 复盘

- 当前分级：操作辅助 + 复盘型界面
- 第一阶段策略：**不要求先做完整新页面**
- 实施方式：优先复用现有页面 + AI 辅助区
- 界面要求：
  - 允许先以辅助面板或现有报告页承接
  - 后续再进入统一复盘页骨架
- 验收要求：
  - 可生成、查看、修订日报/复盘草案
  - AI 建议不冒充正式结论

---

## 3. 首批统一组件绑定

P0 最小闭环阶段，以下组件必须优先接入：

1. 标准表格
2. 标准列表筛选条
3. 状态标签
4. 主按钮 / 次按钮 / 危险按钮
5. 危险确认弹窗
6. 证据卡片
7. 审计时间线
8. AI 建议卡片

绑定优先顺序：

1. 平台订单
2. 样品确认
3. 签收
4. 异常
5. WAES 状态 / 证据
6. 日报 / 复盘

---

## 4. 强约束

1. P0 阶段已能满足闭环的现有功能，不允许仅为“统一外观”新建界面。
2. P0 阶段允许新建界面的节点，只限：
   - 平台订单
   - 样品确认 / 打样签样
   - 签收
   - 异常壳层
3. 所有允许新建的界面，必须直接进入：
   - 统一体验骨架
   - 对话模式与操作模式规范
   - 统一组件与设计令牌体系
4. 未通过模块分级判定的界面，不得进入建设。

---

## 5. 节点级验收

每个 P0 节点的界面实施至少通过以下验收：

1. 业务路径可达
2. 状态可见
3. 证据可见
4. 异常可进入统一处理
5. 风险动作有统一确认
6. AI 建议与业务事实分离

如未通过，不得宣称该节点已完成界面实施收口。
