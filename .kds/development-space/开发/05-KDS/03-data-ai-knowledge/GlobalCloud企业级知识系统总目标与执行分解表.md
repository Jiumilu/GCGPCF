# GlobalCloud 企业级知识系统总目标与执行分解表

日期：2026-06-08
适用范围：GlobalCloud 绿色供应链体系中的知识主存层、LLM Wiki、Brain、WAES 以及与一期验收矩阵、对象目录、事件合同相关的知识域实施工作。
状态：执行控制文档。用于统一目标、阶段、交付物、完成标准与当前进度，不代表系统已运行完成。

## 1. 总目标

完成 `GlobalCloud 绿色供应链体系` 企业级知识系统的边界、状态、验收和真实评测收口，使其达到：

1. 可实施；
2. 可审计；
3. 可验收；
4. 可进入首轮知识域实施验证；
5. 不再出现真源、编制层、引擎层、服务层和治理层角色漂移。

一句话收口：

> 将当前“分层口径基本正确”的知识体系，推进到“状态机明确、验收包明确、评测结论落地、可进入实施验证”的正式状态。

## 2. 目标边界

### 2.1 目标内

本轮总目标包含以下工作：

1. 统一知识系统分层：
   - 知识主存层
   - `LLM Wiki`
   - `Brain`
   - `Brain`
   - `WAES`
2. 建立三条硬状态机：
   - 知识发布状态机
   - 知识 ingest 状态机
   - 知识失效/回滚状态机
3. 建立企业级知识系统一期验收包。
4. 完成 `LLM Wiki` 与 `Brain` 的真实场景评测结论收口。
5. 将状态机与评测结论回压到：
   - 对象目录
   - 事件合同
   - 一期验收矩阵
   - 企业级知识系统实施任务书

### 2.2 目标外

本轮总目标不包含：

1. 知识系统生产上线。
2. 全量运行态修复完成。
3. `Brain` 运行稳定性全部闭环。
4. 业务主账系统状态机改造。
5. AI 写业务事实能力。

## 3. 完成标准

只有以下 5 类条件同时满足，才可认为本目标完成。

### 3.1 分层唯一

必须明确且文档一致：

1. 知识主存层是唯一正式真源。
2. `LLM Wiki` 是知识编制与组织视图。
3. `Brain` 是主知识引擎。
4. `Brain` 是知识服务层。
5. `WAES` 是知识治理与审计层。

### 3.2 状态机完整

必须完成：

1. 发布状态机；
2. ingest 状态机；
3. 失效/回滚状态机；
4. 每个状态具备：
   - 进入条件
   - 退出条件
   - 证据要求
   - WAES 审计点
   - 禁止动作

### 3.3 评测结论落地

必须完成：

1. `LLM Wiki` 与 `Brain` 的真实场景对比结论；
2. 将评测结论反映到知识体系角色分工中；
3. 明确哪些结论是方向成立，哪些仍是运行缺口。

### 3.4 验收包完成

必须完成：

1. 一期知识系统验收包设计；
2. 二期预留验收包边界；
3. 一期验收包与现有一期验收矩阵映射关系。

### 3.5 基线回写完成

必须完成：

1. 对象目录一致性回写；
2. 事件合同一致性回写；
3. 一期验收矩阵一致性回写；
4. README 和实施任务书入口一致性回写。

## 4. 分阶段执行分解

### 阶段 1：分层与边界收口

目标：

1. 统一知识系统各层职责；
2. 消除双主和边界重叠；
3. 固定唯一口径。

主要交付物：

1. `GlobalCloud企业级知识系统冲突点与收口方案.md`
2. `GlobalCloud智能体团队-企业级知识系统实施任务书.md`

当前状态：已完成

### 阶段 2：状态机收口

目标：

1. 建立三条核心状态机；
2. 为每个状态补齐硬字段；
3. 防止将引擎状态误写成正式知识状态。

主要交付物：

1. `GlobalCloud企业级知识系统状态机与验收包设计.md`
2. 发布状态明细表
3. ingest 状态明细表
4. 失效/回滚状态明细表

当前状态：进行中

### 阶段 3：真实场景评测收口

目标：

1. 基于真实仓库和 live 结果对比 `LLM Wiki` 与 `Brain`；
2. 形成对现有知识体系的影响结论；
3. 固定后续实施优先级。

主要交付物：

1. `GlobalCloud绿色供应链体系LLM Wiki与Brain测试评估矩阵.md`
2. 知源专项正式只读预检结论
3. 知源专项运行前样本收集记录
4. 评测影响结论

当前状态：已形成第一轮结论

### 阶段 4：验收包设计

目标：

1. 将状态机压成可执行验收包；
2. 区分设计证据与运行证据；
3. 为一期验收矩阵补知识域硬场景。

主要交付物：

1. 一期知识系统验收包
2. 二期预留验收包
3. 一期验收矩阵知识系统映射

当前状态：未完成

### 阶段 5：基线回写

目标：

1. 将收口后的状态机和验收包压回主基线文档；
2. 确保对象、事件、验收三层一致。

主要交付物：

1. 对象目录更新
2. 事件合同更新
3. 一期验收矩阵更新
4. README 更新

当前状态：未完成

## 5. 当前进度判断

### 5.1 已完成

1. 分层口径已收口；
2. 实施任务书已建立；
3. 冲突点已收口；
4. 真实场景评测已给出第一轮结论；
5. 发布状态明细表与 ingest 状态明细表已形成；
6. 知识系统正文首稿第一版已形成。

### 5.2 未完成

1. 失效/回滚状态明细表未完成；
2. 一期验收包未正式形成；
3. 一期验收矩阵未正式回写；
4. 对象目录与事件合同尚未按状态机二次收紧。

### 5.3 当前阶段

当前处于：

> 阶段 2 向阶段 4 过渡阶段

即：

- 状态机主体已进入成文阶段；
- 但验收包和基线回写尚未完成。

## 6. 下一步动作

下一步按以下顺序执行：

1. 完成 **知识失效/回滚状态明细表**；
2. 组装完成 `GlobalCloud企业级知识系统状态机与验收包设计.md`；
3. 补齐企业知识系统一期验收包；
4. 将验收包压入 `GlobalCloud绿色供应链体系一期验收矩阵.md`；
5. 再回补对象目录与事件合同的状态字段。

## 7. 当前状态纪律

本轮工作当前只能判定为：

- 设计与实施收口进行中
- 可继续进入状态机与验收包设计

当前不得写成：

1. 企业级知识系统已完成；
2. 企业级知识系统已运行完成；
3. `Brain` 已稳定承接主检索面；
4. 知识系统已完成运行态验收；
5. `complete`

## 8. 当前结论

当前整体工作已经从“讨论知识系统该怎么分层”推进到“将知识系统推进到可实施、可验收、可治理状态”的执行阶段。

后续是否能进入真正的知识域实施验证，不取决于再补多少口头原则，而取决于以下动作是否完成：

1. 三条状态机是否真正闭合；
2. 一期验收包是否真正落下；
3. 对象、事件、验收三层是否真正一致。
