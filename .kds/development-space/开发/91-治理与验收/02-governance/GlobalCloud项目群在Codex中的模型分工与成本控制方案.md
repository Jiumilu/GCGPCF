# GlobalCloud 项目群在 Codex 中的模型分工与成本控制方案

日期：2026-06-08
状态：建议稿 v1
适用范围：GlobalCloud 绿色供应链体系相关项目群在 Codex 中的日常设计、治理、实施、同步和控制工作。

## 1. 目标

本方案解决两个问题：

1. 在 Codex 中，哪些任务应该优先使用高质量模型。
2. 在不明显牺牲质量的前提下，如何控制整体使用成本。

本方案不追求“全程最强模型”，也不采用“全程最低成本模型”，而是采用分层用模。

## 2. 总体结论

推荐采用三层模型分工：

1. 主质量模型：`GPT-5.2-Codex`
2. 日常成本模型：`GPT-5 mini`
3. 边角低成本模型：`GPT-5 nano`

核心原则：

1. 影响架构、边界、状态、治理、代码主线的任务，优先使用 `GPT-5.2-Codex`。
2. 模板化、重复性、结构明确的任务，优先使用 `GPT-5 mini`。
3. 极轻量、非关键、机械型任务，才考虑 `GPT-5 nano`。

## 3. 模型分工

### 3.1 GPT-5.2-Codex

定位：

- 高质量主线模型
- 长链路编码与跨项目治理模型

适用任务：

1. 体系架构设计
2. 跨模块重构
3. 多文件代码修改
4. 复杂调试
5. 项目群治理
6. 关键文档定版
7. 真实运行问题定位
8. 高风险实施判断

不适合的使用方式：

1. 大量重复性台账抄写
2. 简单周报格式化
3. 大量机械清单更新

建议推理档位：

1. 日常复杂任务：`medium`
2. 关键设计与复杂改造：`high`
3. 特别复杂、跨域、长链条问题：按需提升，但不默认全程最高档

### 3.2 GPT-5 mini

定位：

- 日常成本控制主力模型
- 结构明确任务的默认副模型

适用任务：

1. 周报整理
2. Linear 同步文案
3. 模板化文档补齐
4. 低风险清单维护
5. 固定格式回写
6. 常规状态汇总
7. 规则明确的文本整理

不适合的使用方式：

1. 项目群顶层架构取舍
2. 复杂代码主线重构
3. 多系统边界裁决
4. 关键运行问题最终定因

建议使用原则：

1. 先判断任务是否“结构清晰、边界固定、低风险”。
2. 若是，则默认优先 `GPT-5 mini`。
3. 若出现反复返工、判断不稳、跨域冲突，再升级到 `GPT-5.2-Codex`。

### 3.3 GPT-5 nano

定位：

- 最低成本辅助模型

适用任务：

1. 简单标题整理
2. 纯格式转换
3. 极轻量摘要
4. 批量机械分类

不适合的使用方式：

1. 架构设计
2. 项目治理
3. 实施计划编排
4. 状态升级判断
5. 代码与运行问题分析

结论：

`GPT-5 nano` 不应成为你这批项目的主力模型。

## 4. 项目群模型建议

| 项目/域 | 推荐主模型 | 推荐副模型 | 说明 |
|---|---|---|---|
| WAES | `GPT-5.2-Codex` | `GPT-5 mini` | 治理、证据、状态、授权边界要求高 |
| PVAOS | `GPT-5.2-Codex` | `GPT-5 mini` | 涉及入口、组织、伙伴、项目边界 |
| GPC / GPC | `GPT-5.2-Codex` | `GPT-5 mini` | 供应链协同主线，不适合低质量裁决 |
| GFIS | `GPT-5.2-Codex` | `GPT-5 mini` | 工厂事实主账、执行主链要求高 |
| Edge | `GPT-5.2-Codex` | `GPT-5 mini` | 接入、安全、补传、回执边界敏感 |
| KDS / Brain / 知识引擎 | `GPT-5.2-Codex` | `GPT-5 mini` | 知识边界、引用、失效治理要求高 |
| XiaoC / Hermes / XGD | `GPT-5.2-Codex` | `GPT-5 mini` | AI 授权、Agent 权限、越权拦截需要高质量判断 |
| 小即总控 / PMBOK / 控制塔 | `GPT-5.2-Codex` | `GPT-5 mini` | 总控与治理结论应以高质量为主 |
| Linear 周报 / 台账同步 | `GPT-5 mini` | `GPT-5 nano` | 结构明确，适合低成本处理 |
| 机械型清单整理 | `GPT-5 mini` | `GPT-5 nano` | 可进一步压低成本 |

## 5. 最佳使用模型

如果只给一个可执行结论：

### 最佳组合

1. **关键质量主线**：`GPT-5.2-Codex`
2. **日常成本副线**：`GPT-5 mini`

### 不建议的组合

1. 全部任务都用 `GPT-5.2-Codex`
   - 成本过高
   - 对模板化任务浪费明显

2. 全部任务都用 `GPT-5 mini`
   - 架构、治理、复杂调试和跨域裁决质量风险偏高

3. 大量主线任务用 `GPT-5 nano`
   - 风险不可接受

## 6. 任务分配策略

建议按以下顺序判断模型：

### 第一类：必须高质量

直接使用 `GPT-5.2-Codex`

包括：

1. 架构设计
2. 跨项目治理
3. 多文件代码修改
4. 复杂调试
5. 状态裁决
6. 关键文档定版

### 第二类：优先成本控制

先使用 `GPT-5 mini`

包括：

1. 周报
2. 台账
3. 模板化文档
4. 固定格式同步
5. 常规结构化整理

### 第三类：极轻量任务

按需使用 `GPT-5 nano`

包括：

1. 标题改写
2. 简单摘要
3. 纯格式处理

## 7. 成本控制策略

### 7.1 不要把最强模型用于机械工作

高质量模型只用在：

1. 需要判断的地方
2. 需要推理的地方
3. 需要跨域整合的地方

### 7.2 先用 mini，失败再升级

适用于：

1. 文档补齐
2. 清单更新
3. 周报台账同步

若出现：

1. 反复返工
2. 结论不稳
3. 结构错误
4. 边界混乱

再升级到 `GPT-5.2-Codex`。

### 7.3 不把低成本模型用于治理裁决

以下任务默认禁止交给 `GPT-5 nano`，并谨慎交给 `GPT-5 mini`：

1. 是否升级状态
2. 是否具备验收条件
3. 是否属于真实完成
4. 是否允许跨系统边界
5. 是否允许 AI 具备某类权限

## 8. 对当前项目群的直接建议

对你当前左侧项目群，最实用的使用模型是：

1. `GPT-5.2-Codex` 负责：
   - WAES / GPC / GFIS / Brain / XiaoC / Hermes / 控制塔 / PMBOK 主线
2. `GPT-5 mini` 负责：
   - Linear 同步
   - 周报
   - 模板回写
   - 低风险台账维护
3. `GPT-5 nano` 仅负责：
   - 极轻量格式处理
   - 批量机械性整理

## 9. 最终结论

左侧项目从质量和成本角度，在 Codex 中可使用的最佳使用模型不是单模型，而是：

**`GPT-5.2-Codex` + `GPT-5 mini` 的双主力模型方案。**

其中：

1. `GPT-5.2-Codex` 保证质量
2. `GPT-5 mini` 控制成本
3. `GPT-5 nano` 只做边角任务

这套方案最适合当前 `GlobalCloud 绿色供应链体系` 这类多项目、多边界、多治理约束的工作场景。
