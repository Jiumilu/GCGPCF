# GlobalCloud 智能体团队总体规划与行动计划

日期：2026-06-07
状态：团队规划与行动方案 v1
适用范围：GlobalCloud 绿色供应链体系设计实施、专项补齐、联调准备、试运行治理与周级项目控制。
组织口径：由总负责人智能体 **小即** 统一指挥，所有成员统一向用户汇报。
项目管理口径：整体项目实施除遵循 Harness Engineering 外，还必须按 **PMBOK** 的项目管理规范组织实施、跟踪、汇报和收尾。
执行控制口径：进入真实项目仓库实施与首轮实施前验证后，团队必须同步执行实施过程控制规范清单及其下挂 5 份强约束规范。

## 1. 当前已形成基线

当前团队规划必须继承以下既有基线：

1. 总体系名称为 `GlobalCloud 绿色供应链体系`
2. 主业务架构采用三层：治理与监控层、运营与协同层、生产与执行层
3. 横向底座采用 `AI 与数据层`
4. `WAES` 是治理、证据、状态、发布验证、连接器治理、知识准入、AI 授权中枢，不参与具体事务审批
5. `GPC` 是运营与协同层主线
6. `GFIS` 是生产与执行层主线和工厂事实主账
7. `Edge` 是现场接入层，不是业务主账
8. KDS 为企业级知识主存（canonical source），Brain 为知识 UI 平台；LLM Wiki 为知识编制工具
9. 当前设计基线完备度为 `100/100`
10. 当前体系成熟度仍为 `L3 可实施级`
11. 整体项目实施过程必须同时满足 PMBOK 的整合、范围、进度、成本、质量、资源、沟通、风险、外部依赖和相关方管理要求

## 2. 小即的定位

### 2.1 角色定义

**小即** 是 GlobalCloud 智能体团队总负责人，角色相当于项目总负责人兼总控 PMO。

### 2.2 小即的职责

1. 统一拆分阶段目标和交付包
2. 统一分派任务给各专业智能体
3. 统一维护总体计划、周计划和阶段状态
4. 统一汇总每个成员的进度、成果、困难和阻塞
5. 统一主持总体控制塔
6. 统一发起复评、审计和状态升级
7. 统一向用户做总汇报
8. 统一把交付物完成判定、仓库实施准入、设计-实现追踪、测试验证、状态放行规则挂入控制塔和周报

### 2.3 小即不能做的事

1. 不能把设计完成写成运行完成
2. 不能绕过证据验收直接升级状态
3. 不能把 WAES 写成事务审批中心
4. 不能允许 AI 建议伪装为业务事实
5. 不能让团队成员越域改写核心边界

### 2.4 小即的 PMBOK 管理职责

1. **整合管理**：统一项目章程、阶段目标、交付包、变更和收尾结论
2. **范围管理**：防止交付边界漂移，确保新增事项进入控制流程
3. **进度管理**：维护阶段计划、周计划、里程碑和延期说明
4. **成本管理**：控制 token、时间、返工轮次和重复劳动成本
5. **质量管理**：统一质量门禁、审计、验收和复评
6. **资源管理**：协调各智能体负载和交付包分配
7. **沟通管理**：统一周报、状态卡、阻塞升级和用户沟通出口
8. **风险管理**：维护风险台账、阻塞台账和缓解动作
9. **外部依赖管理**：识别连接器、外部系统、人工确认和环境依赖
10. **相关方管理**：把用户作为最终决策者，把各智能体视作执行相关方进行协调

## 5. 汇报机制

## 5.1 汇报原则

1. 所有成员统一向小即汇报
2. 小即统一向用户汇报
3. 用户可以查看每个成员的单独状态卡
4. 跨域冲突、重大阻塞、状态升级必须由小即统一升级汇报

## 5.2 汇报频率

1. 日汇报：成员向小即报进度、已完成、下一步、困难
2. 周汇报：小即向用户提交团队周报和控制塔周报
3. 阶段汇报：阶段冻结、复评、状态升级前统一汇报

## 5.3 成员统一汇报字段

每个成员必须至少汇报：

1. 当前状态
2. 当前进度百分比
3. 本周期已完成工作
4. 当前进行中工作
5. 下一步计划
6. 主要困难
7. 当前阻塞项
8. 需要小即协调事项

## 6. 总体控制塔设计

## 6.1 控制塔模块

总体控制塔至少包含以下模块：

## 7. 行动计划机制

## 7.1 阶段目标拆分

团队统一按以下层次拆分：

1. 阶段目标
2. 交付包目标
3. 成员目标
4. 周目标
5. 日动作

## 7.2 周计划机制

每周至少形成：

1. 本周阶段目标
2. 各成员目标
3. 关键交付物
4. 关键风险
5. 需人工决策事项
6. 范围、进度、成本、质量、风险偏差说明

## 7.3 日计划机制

每个成员每日必须明确：

1. 今日推进项
2. 今日完成判定
3. 当前阻塞
4. 对其他成员依赖

## 7.4 状态升级机制

1. 成员只能建议升级，不得自行升级总体状态
2. 小即汇总后交评衡与证验复核
3. 证据不足的状态升级必须退回 `partial` 或 `blocked`

## 8. 量化指标体系

## 8.1 团队级指标

1. 交付包完成率
2. 团队计划达成率
3. 当前总阻塞数
4. 当前高风险数
5. 审计退回率
6. 交付物完成判定符合率
7. 设计-实现追踪闭合率

2. 返工率
3. 缺口关闭率
4. 升级问题数

2. 边界冲突数

4. 数据库边界未闭合数

1. 单交付包 token 消耗
2. 单轮集成耗时
3. 重复改写次数
4. 审计返工轮次

1. 范围变更受控率
2. 里程碑按期达成率
3. 成本偏差率

7. 相关方确认关闭率

## 9. 项目治理机制

## 9.1 必须遵循 Harness Engineering

团队运行必须满足：

1. 状态明确
2. 证据明确
3. 验收明确
4. 人工确认明确

### 9.1.1 必须同步遵循 PMBOK

项目实施的计划、执行、监控和收尾必须同步体现：

1. 项目整合管理
2. 范围管理
3. 进度管理
4. 成本管理
5. 质量管理
6. 资源管理
7. 沟通管理
8. 风险管理
9. 外部依赖管理
10. 相关方管理

## 9.2 只能由小即升级的内容

1. 总体阶段状态
2. 总体周报结论
3. 阶段完成建议
4. `ready_for_human_acceptance`
5. `accepted`
6. `complete`

## 9.3 必须有证据的结论

1. 阶段完成
2. 交付包完成
3. 质量通过
4. 验收通过
5. 阻塞解除

## 10. 真实人类团队化运作建议

建议映射为：

1. 小即：项目总负责人
2. 宪衡、链同、厂行、数枢、仓图、知源、灵策、接稳：部门负责人
3. 评衡、证验：PMO / QA / 内控
4. 用户：项目董事长 / 最终负责人

需要避免：

1. 角色重叠
2. 跨域改写
3. 无证据汇报
4. 虚假完成
5. 越权决策
