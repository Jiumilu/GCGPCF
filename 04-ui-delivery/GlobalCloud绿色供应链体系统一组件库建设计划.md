# GlobalCloud 绿色供应链体系统一组件库建设计划

日期：2026-06-08
状态：统一组件库建设计划 v1
用途：把统一组件与设计令牌规范转成实际组件库建设步骤、阶段目标、范围和优先级。

---

## 1. 建设目标

建设一套面向 GlobalCloud 绿色供应链体系最终交付的统一组件库，覆盖：

1. 基础设计令牌
2. 基础交互组件
3. 数据展示组件
4. 状态与反馈组件
5. 审计与证据组件
6. AI 专用组件
7. 页面模板组件

目标不是做一套“好看的 UI 库”，而是做一套能够让 WAES、PVAOS、GPC、GFIS、Brain、XiaoC（蚁后）、Hermes/XGD（大象） 最终长成同一产品体系的实施底座。

---

## 2. 建设原则

1. **先满足 P0 闭环，再追求全量覆盖**
2. **先做高复用组件，再做复杂业务组件**
3. **先做结构和语义统一，再做视觉细节统一**
4. **先做组件令牌，再做主题包装**

---

---

## 4. 阶段划分

### 阶段 A：令牌与基础组件

目标：

1. 固定颜色、字体、间距、圆角、阴影、边框令牌
2. 产出基础组件
3. 让 P0 最小闭环能够开始共用组件

### 阶段 B：结构组件

目标：

1. 固定列表、详情、配置、工作台、异常、证据、AI 的模板组件
2. 让高频页面不再自由生长

### 阶段 C：多端与主题层

目标：

1. 收口 Web / 桌面 / 移动 / 对话端差异
2. 形成最终交付风格一致性

---

6. 证据卡片

### P1 优先建设

1. 工作台模板
2. 异常页模板
3. 审计时间线
4. 详情页模板

### P2 优先建设

1. 图标系统
2. 品牌主题层
3. 动效系统

---

## 6. 与实施分级判定的关系

组件库建设必须服务于模块分级结果：

1. **直接可用**：尽量不改业务逻辑，只套统一组件
2. **产品内实现**：优先用现有配置、模板、脚本接入统一组件
3. **二次开发**：按统一组件补足功能
4. **局部开发**：旁路实现也必须复用统一组件
5. **全面重构**：重构时必须以组件库为底座

---

## 7. 当前阶段结论

当前最需要的不是再补一套抽象设计原则，而是：

1. 明确首批组件清单
2. 明确高风险模块优先接入哪些组件
3. 明确组件库建设与业务实施如何同步推进

因此，这份计划的意义是把“统一设计系统”从概念变成建设任务。
