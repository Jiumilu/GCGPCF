# GlobalCloud 绿色供应链体系统一组件与设计令牌规范

日期：2026-06-08  
状态：统一组件与设计令牌规范 v1  
适用范围：WAES、PVAOS、GPC-Native、GFIS、Brain、XiaoC、Hermes、XGD，以及最终交付中的 Web、桌面端、移动端、消息端界面。  
目的：把前面的界面分阶段治理规则、统一体验骨架规范、最终交付界面规范与样式规范、对话模式与操作模式规范，进一步收口为可落地的设计系统约束，确保最终交付产品在颜色、字体、间距、图标、组件、状态标签、弹窗、表格、表单等方面形成统一产品语言。

---

## 1. 总体原则

本规范只解决两类问题：

1. **设计令牌统一**
   - 颜色
   - 字体
   - 间距
   - 圆角
   - 阴影
   - 边框
   - 尺寸

2. **组件统一**
   - 按钮
   - 输入框
   - 表单
   - 表格
   - 状态标签
   - 卡片
   - 弹窗
   - 抽屉
   - Toast
   - 空态/错误态/加载态

原则：

1. 同一语义必须使用同一设计令牌
2. 同一操作必须使用同一组件样式
3. 同一状态必须使用同一颜色与标签规范
4. 不允许各系统自行派生不同风格体系
5. 不允许先做一套页面，再事后补做“统一样式”

---

## 2. 设计令牌体系

### 2.1 颜色令牌

最终统一以下颜色组：

1. 主色 `brand-primary`
2. 辅色 `brand-secondary`
3. 背景色 `bg-*`
4. 文本色 `text-*`
5. 语义色：
   - `success`
   - `warning`
   - `danger`
   - `info`
   - `muted`
6. 状态色：
   - `status-running`
   - `status-blocked`
   - `status-partial`
   - `status-review`
   - `status-complete`

约束：

1. 业务状态色与治理状态色必须共享语义体系
2. 危险操作只能使用 `danger` 体系
3. AI 建议不得使用和业务完成相同的状态色
4. 治理阻断不得使用普通说明色

### 2.2 字体令牌

统一定义：

1. 标题字体
2. 正文字体
3. 等宽字体
4. 字号层级：
   - `display`
   - `title-1`
   - `title-2`
   - `title-3`
   - `body`
   - `body-small`
   - `caption`

约束：

1. 不允许不同系统单独定义标题层级
2. 表格、表单、标签、说明文字必须共用同一字号体系

### 2.3 间距令牌

统一定义：

- `space-2`
- `space-4`
- `space-8`
- `space-12`
- `space-16`
- `space-24`
- `space-32`

约束：

1. 页面外边距统一
2. 组件内边距统一
3. 表单项间距统一
4. 列表与卡片间距统一

### 2.4 圆角、边框、阴影令牌

统一定义：

1. 圆角：
   - `radius-sm`
   - `radius-md`
   - `radius-lg`
2. 边框：
   - `border-default`
   - `border-muted`
   - `border-danger`
3. 阴影：
   - `shadow-sm`
   - `shadow-md`
   - `shadow-lg`

约束：

1. 业务型后台界面不允许使用夸张圆角
2. 不允许组件随意使用不同阴影层级
3. 卡片、抽屉、弹窗应共享同一阴影体系

---

## 3. 组件规范

### 3.1 按钮

统一按钮类型：

1. 主按钮
2. 次按钮
3. 线框按钮
4. 危险按钮
5. 纯文本按钮
6. 图标按钮

统一要求：

1. 相同优先级动作使用相同按钮类型
2. 危险操作必须使用危险按钮
3. 主按钮一页只允许一个主视觉优先级
4. 图标按钮必须有清晰 Tooltip

### 3.2 输入组件

统一组件：

1. 单行输入
2. 多行输入
3. 下拉选择
4. 多选
5. 日期选择
6. 时间选择
7. 组合选择器
8. 搜索框
9. 标签输入
10. 文件上传

统一要求：

1. 校验反馈位置统一
2. 必填标识统一
3. 占位文案写法统一
4. 禁用态样式统一

### 3.3 表格

统一表格规范：

1. 表头样式统一
2. 行高统一
3. 排序样式统一
4. 筛选样式统一
5. 操作列样式统一
6. 状态列展示统一

统一要求：

1. 同类对象列表不得采用两套表格交互
2. 表格中的危险操作必须降到菜单或二次确认

### 3.4 状态标签

统一标签类型：

1. 业务状态标签
2. 治理状态标签
3. 风险标签
4. AI 标签
5. 异常标签

统一要求：

1. 颜色和语义必须固定
2. 相同状态文案不能在不同系统写成不同词
3. AI 建议标签必须显式区分“建议”与“事实”

### 3.5 卡片

统一适用范围：

1. 摘要卡片
2. 指标卡片
3. 异常卡片
4. 证据卡片

统一要求：

1. 卡片不允许变成页面主要容器风格泛滥
2. 卡片只用于摘要、局部聚合或重复项

### 3.6 弹窗与抽屉

统一要求：

1. 弹窗用于短确认、短编辑、风险提示
2. 抽屉用于上下文保留型编辑或查看
3. 危险确认弹窗必须固定结构：
   - 影响对象
   - 风险说明
   - 确认动作
   - 取消动作

### 3.7 Toast / 消息提示

统一类型：

1. 成功
2. 警告
3. 错误
4. 信息

统一要求：

1. 文案模板统一
2. 不允许把治理阻断写成普通成功/说明提示
3. 错误提示必须可定位下一步

### 3.8 空态 / 加载态 / 错误态

统一要求：

1. 空态必须说明对象为空还是当前筛选为空
2. 加载态必须区分首次加载与局部刷新
3. 错误态必须区分权限、网络、治理阻断、系统异常

---

## 4. AI 组件规范

统一 AI 组件：

1. AI 对话区
2. AI 建议卡片
3. AI 引用区
4. AI 风险提示条
5. AI 驳回/采纳操作栏

统一要求：

1. AI 组件必须有独立视觉标识
2. AI 输出不得与系统事实卡片长得完全一样
3. AI 建议卡片必须提供来源和风险说明

---

## 5. 多端组件统一

必须统一的组件语义：

1. 按钮优先级
2. 状态标签
3. 表单校验
4. 风险确认
5. 证据查看入口
6. AI 建议卡片

允许差异的部分：

1. 导航形态
2. 布局密度
3. 移动端折叠方式
4. 消息端文本化表达

但不允许差异到改变：

1. 状态语义
2. 风险语义
3. 治理语义
4. 确认逻辑

- 对话模式与操作模式规范：决定怎么交互
- 本规范：决定组件和令牌怎么统一落地

## 7. 验收要求

以下内容必须纳入最终 UI/UX 验收：

1. 颜色是否统一
2. 字体层级是否统一
3. 间距是否统一
4. 按钮优先级是否统一
5. 表单样式是否统一
6. 表格交互是否统一
7. 状态标签语义是否统一
8. 危险操作确认样式是否统一
9. AI 建议组件是否统一
10. 多端组件语义是否统一

未通过上述验收，不得宣称设计系统已统一。

当前已经具备：

1. 分阶段界面治理规则
2. 统一体验骨架规范
3. 最终交付界面规范与样式规范
4. 对话模式与操作模式规范
5. 统一组件与设计令牌规范

当前仍未自动成立的是：

1. 各系统是否已按该规范完成组件收口
2. 是否已经存在统一组件库

因此，本规范是产品设计系统层的硬约束，不代表当前实现已全部达标。
