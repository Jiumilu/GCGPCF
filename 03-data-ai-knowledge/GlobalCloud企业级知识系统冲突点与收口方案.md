# GlobalCloud 企业级知识系统冲突点与收口方案

日期：2026-06-08  
适用范围：GlobalCloud 绿色供应链体系中的企业级知识主存域、知识引擎域、KDS、Brain、LLM Wiki、WAES 治理域、AI 服务域。  
状态：设计收口文档。用于统一当前口径，不代表真实运行完成。

## 1. 目的

当前企业级知识系统主方向已经基本一致，但在实施落地层面仍存在若干角色重叠、表述漂移和承载不唯一的问题。  
本文件用于：

1. 明确当前存在的冲突点。
2. 给出最终唯一口径。
3. 为后续对象目录、事件合同、验收矩阵、知识治理模型和专项实施任务书提供统一约束。

## 2. 当前已形成且不应再回退的基线

以下口径当前已经成立，应视为已冻结基线：

1. 企业级知识真源必须位于 **知识主存层**。
2. `LLM Wiki` 与 `Brain` 当前都不是企业级知识真源。
3. `WAES` 只负责知识准入、证据分级、引用审计、发布验证和 AI 授权，不做知识正文主账。
4. AI 建议、摘要、候选知识条目不能直接改写正式知识版本。
5. 正式发布链必须是单向链路：**知识主存层 -> 知识引擎层 -> 消费层**。

## 3. 当前存在的主要冲突点

### 3.1 Brain 定位仍有轻微重叠

当前文档中，`Brain` 同时被描述为：

1. 知识引擎层组成部分；
2. 知识服务层；
3. 面向 AI 和业务应用的知识域编排层。

这会导致一个实施冲突：

- 一旦 `Brain` 同时在知识引擎层和知识服务层承担重叠职责，就会与 `LLM Wiki` 的编制层边界模糊，形成双主风险。

### 3.2 LLM Wiki 与 Brain 的关系尚未完全唯一化

当前文档已经隐含以下分工：

1. `LLM Wiki` 偏知识编制、知识组织和规范表达。
2. `Brain` 偏索引、检索、RAG、图谱和统一引擎能力。

但当前仍有一个实现层冲突：

- 到底是“双引擎并行长期存在”，还是“一个编制层 + 一个主知识引擎”。

如果不明确，后续实施时容易出现双主。

### 3.3 知识主存层已经被要求存在，但承载体还未完全唯一化

当前设计已经要求必须有独立知识主存层，但仍未完全固定：

1. 最终由什么系统承载正式知识主存；
2. 文档主存、结构化元数据、发布链、权限链、归档链之间如何一一绑定；
3. 是否作为独立产品/独立存储域落地。

这不是原则冲突，但属于实现承载冲突。

### 3.4 WAES 与 Brain 在知识域中的边界仍需再收紧

虽然当前已经明确 `WAES` 不做正文主账，但在实施中仍可能被误读为：

1. `WAES` 直接管理知识正文；
2. `Brain` 也直接管理知识正文；
3. 两者都能成为知识内容控制中心。

如果不收口，后续会出现治理平面与内容平面混淆。

### 3.5 发布链、失效链、引用链原则已明确，但状态机尚不够硬

当前已经提出：

1. 知识需要发布；
2. 需要失效治理；
3. 需要引用审计；
4. 需要 AI 引用约束。

但仍缺少更硬的统一状态机，例如：

- 草稿
- 评审中
- 已发布
- 已失效
- 已归档
- 已同步到引擎
- 引擎同步失败
- AI 禁止引用

如果没有统一状态机，实施时会出现不同模块各自定义状态。

## 4. 最终收口口径

### 4.1 最终层次分工

企业级知识系统的最终唯一口径应为：

1. **KDS（知识主存层）**：唯一正式知识真源。
2. **LLM Wiki**：知识编制与组织视图。

3. **Brain**：主知识引擎 + 知识服务层（索引/检索/RAG/图谱/SOP/案例/复盘/消费编排）。
4. **WAES**：知识治理与审计层。

### 4.2 各角色唯一职责

#### KDS（知识主存层）

只负责：

1. 正式知识文档；
2. 版本；
3. 发布；
4. 权限；
5. 归档；
6. 审计元数据。

不负责：

1. AI 推理；
2. RAG 服务；
3. 业务交互服务；
4. 知识服务编排。

#### LLM Wiki

只负责：

1. 知识编制视图；
2. 章节组织；
3. 规范展示；
4. 人工整理和维护友好的知识页面。

不负责：

1. 企业正式知识真源；
2. 主引擎唯一检索能力；
3. 反写知识主存层。

#### Brain（主知识引擎 + 知识服务层）

只负责：

1. 主知识引擎：索引、检索、RAG、图谱、引用回指；
2. 知识服务层：SOP / 案例 / 复盘等知识域服务；
3. 面向 AI、控制塔、业务应用的知识消费编排；
4. 对外统一知识服务接口。

不负责：

1. 正式知识真源；
2. 正式版本发布主账；
3. 正式文档直接发布。

#### WAES

只负责：

1. 知识准入；
2. 证据等级；
3. 引用审计；
4. 发布验证；
5. AI 授权；
6. 失效拦截；
7. 治理结论。

不负责：

1. 知识正文主账；
2. 知识正文编写；
3. 事务审批；
4. 业务主账确认。

## 5. 最终协同链路

正确链路应固定为：

**知识主存层**  
-> 正式发布  
-> `KnowledgeIngestJob`  
-> `LLM Wiki` 编制视图  
-> `Brain` 知识引擎与服务层  
-> `XiaoC / Hermes / XGD` 消费  
-> `WAES` 审计、授权和治理确认

## 6. 明确禁止的冲突性做法

以下做法应明确禁止：

1. 把 `LLM Wiki` 写成企业级知识真源。
2. 把 `Brain` 写成企业级知识真源。
3. 允许 `LLM Wiki` 反写知识主存层。
4. 允许 `Brain` 直接修改主存层正式文档。
5. 允许 `Brain` 直接替代知识主存层发布正式版本。
6. 把 `WAES` 写成知识正文主账。
7. 允许 AI 直接修改正式知识版本。
8. 在实施方案中同时把 `LLM Wiki` 和 `Brain` 都写成主知识引擎而不说明主次。

## 7. 建议补齐的状态机

建议后续统一采用以下知识生命周期状态：

1. `draft`
2. `in_review`
3. `approved_for_release`
4. `released`
5. `deprecated`
6. `archived`
7. `ingest_pending`
8. `ingest_completed`
9. `ingest_failed`
10. `ai_restricted`

## 9. 当前结论

当前企业级知识系统**没有大的方向性冲突**，但在实施层面仍存在以下 5 个需要收口的问题：

1. `Brain` 定位重叠；
2. `LLM Wiki` 与 `Brain` 主次尚未完全唯一化；
3. 知识主存层承载体未完全唯一化；
4. `WAES` 与 `Brain` 在知识域边界仍需收紧；
5. 知识发布/失效/引用状态机仍需硬化。

最终建议唯一口径：

- **KDS（知识主存层）**：唯一正式真源  
- **LLM Wiki**：知识编制与组织视图  
- **Brain**：主知识引擎 + 知识服务层  
- **WAES**：知识治理与审计层
