# GlobalCloud 绿色供应链体系 LLM Wiki 与 Brain 测试评估矩阵

日期：2026-06-07
状态：评估矩阵 v1
适用范围：`LLM Wiki`、`Brain`、`WAES`、`KDS`（知识主存层设计）。

## 1. 评估目标

本矩阵不用于判断谁是企业级知识库真源，而是用于判断：

1. 谁更适合作为知识引擎层主线。
2. 谁更适合作为知识编制或知识索引能力。
3. 在企业级知识主存层之上，两者各自需要补哪些能力。

## 2. 评估原则

1. 不把“能检索”当成“企业级可落地”。
2. 不把“内容多”当成“治理成熟”。
3. 不把“RAG 可用”当成“正式知识发布能力”。
4. 不把“AI 输出好”当成“可替代知识真源”。

## 3. 评估维度

| 维度 | 说明 | 权重 |
|---|---|---:|
| 知识组织能力 | 目录、空间、跨主题结构化表达 | 10 |
| 元数据能力 | 编号、标签、版本、来源、分类 | 10 |
| 权限隔离能力 | 项目、组织、密级、AI 可见性 | 12 |
| 发布治理能力 | 草稿、评审、批准、生效、归档 | 12 |
| RAG 检索质量 | 命中率、回指、召回、噪声控制 | 12 |
| 引用可信度 | `source_refs`、切片回指、引用稳定性 | 10 |
| Agent 消费适配 | Brain / XiaoC / Hermes / XGD 的接入效率 | 8 |
| 同步与更新能力 | 增量 ingest、重建、失效、删除 | 8 |
| 审计能力 | 查询、引用、发布、导出日志 | 8 |
| 运维与恢复能力 | 备份、恢复、重建、迁移 | 5 |
| 多租户与多项目能力 | 空间隔离、跨项目并行 | 5 |

总分：100

## 4. 建议测试项

### 4.1 知识组织

1. 能否稳定表达宪法、体系、葛化项目、SOP、政策、复盘等不同空间。
2. 能否做跨空间但不越权的统一检索。

### 4.2 引用可信度

1. 输出是否稳定带 `source_refs`。
2. 是否能回指到文档、标题、段落、切片。
3. 是否支持失效文档引用拦截。

### 4.3 权限与隔离

1. 能否按项目隔离。
2. 能否按空间隔离。
3. 能否按角色或密级限制。

### 4.4 更新能力

1. 文档更新后索引刷新是否准确。
2. 已删除或退役文档是否能被及时失效。
3. 重建后引用是否保持稳定。

### 4.5 AI 消费能力

1. Brain 消费是否稳定。
2. Hermes/XGD 问答和 Agent 调用是否稳定。
3. 是否容易把未确证内容误当强事实。

## 5. 升级判定

### 5.1 可作为主知识引擎的最低门槛

1. 总分不低于 80。
2. `引用可信度` 不低于 8/10。
3. `权限隔离能力` 不低于 8/12。
4. `同步与更新能力` 不低于 6/8。
5. `审计能力` 不低于 6/8。

### 5.2 不得直接作为企业级知识真源的判定

任一项成立即不应直接作为企业级知识真源：

1. 无正式版本链。
2. 无发布审批链。
3. 无组织和密级权限。
4. 无知识审计日志。
5. 无归档与保留策略。

## 6. 升级建议输出格式

每轮测试后建议输出：

1. 当前适配角色：
   - 编制层 / 检索层 / 主引擎 / 迁移工具
2. 当前主要短板：
   - 权限 / 发布 / 审计 / 引用 / 隔离 / 运维
3. 是否满足知识引擎主线门槛。
4. 距离企业级知识库还缺什么。

## 7. 当前建议

现阶段建议：

1. `LLM Wiki` 重点验证内容编制、知识组织和规范连续性。
2. `Brain` 重点验证索引、RAG、引用回指和多空间隔离。
3. 企业级知识主存层独立设计，不以任一测试结果直接替代。
