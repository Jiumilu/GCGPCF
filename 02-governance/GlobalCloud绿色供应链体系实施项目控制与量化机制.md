# GlobalCloud 绿色供应链体系实施项目控制与量化机制

日期：2026-06-07  
状态：实施控制模型 v1  
目标：为 GlobalCloud 绿色供应链体系建立可量化、可审计、可控成本、可控质量的实施控制机制。  
执行基线：本机制与交付物完成判定、项目仓库实施准入、设计-实现追踪、测试与验证、状态升级与验收放行五类过程控制规范绑定执行。

## 1. 控制目标

本机制用于同时控制：

1. 交付进度
2. 设计质量
3. 证据与状态纪律
4. token、时间与返工成本
5. 跨域冲突和阻塞

## 2. 交付包控制对象

当前按 12 个交付包进行项目控制：

1. WAES 治理与控制塔包
2. PVAOS 生态入口包
3. GPC-Native 运营协同包
4. GFIS 工厂执行包
5. Edge 边缘接入包
6. AI 与数据底座包
7. 企业级知识主存与知识引擎包
8. XiaoC + Hermes + XGD AI 服务包

## 3. 项目控制看板

建议对每个交付包固定维护如下字段：

| 字段 | 说明 |
|---|---|
| 交付包 | 当前工作包名称 |
| 主责智能体 | 当前牵头角色 |
| 当前状态 | Harness 状态标签 |
| 当前得分 | 对应评估分值或阶段得分 |
| 缺口数 | 当前未闭合关键缺口数 |
| 阻塞项 | 当前阻塞原因 |
| 下一步 | 下一条明确执行动作 |
| 需人工确认 | 是否需要人工准入或验收 |

## 4. 量化指标体系

### 4.1 进度指标

| 指标 | 定义 | 目标 |
|---|---|---|
| 交付包完成率 | 已达阶段目标的交付包 / 全部交付包 | 阶段内逐步提升 |
| 对象补齐率 | 已定义对象 / 计划对象 | 100% |
| 事件补齐率 | 已定义事件 / 计划事件 | 100% |
| 验收场景补齐率 | 已定义验收场景 / 计划场景 | 100% |
| 控制文档覆盖率 | 已形成控制文档 / 应有控制文档 | 100% |

### 4.2 质量指标

| 指标 | 定义 | 目标 |
|---|---|---|
| 设计基线得分 | 当前设计基线评分 | 100/100 |
| 边界冲突数 | 主责、对象、事件、数据库冲突数量 | 0 |
| AI 越权缺口数 | AI 仍可越权的未闭合风险项 | 0 |
| 数据库边界未闭合数 | 共享主账、跨域直写等未闭合项 | 0 |
| 审计退回率 | 被审计退回的交付项 / 总交付项 | 持续下降 |

### 4.3 治理指标

| 指标 | 定义 | 目标 |
|---|---|---|
| 状态升级准确率 | 符合证据门禁的状态升级 / 全部升级 | 100% |
| 证据完整率 | 可回指证据的结论 / 全部结论 | 100% |
| 未经审计定版次数 | 无审计直接定版次数 | 0 |
| 驳回后返工率 | 被驳回后需重做项 / 全部项 | 持续下降 |

### 4.4 成本指标

| 指标 | 定义 | 控制方向 |
|---|---|---|
| 单交付包 token 消耗 | 单交付包处理的 token 使用量 | 持续监控 |
| 单轮集成耗时 | 从专项提交到总控集成耗时 | 持续压缩 |
| 文档重复改写次数 | 同一主文档被重复大幅改写次数 | 最小化 |
| 审计返工轮次 | 审计退回导致的重复轮次 | 最小化 |
| 交付物完成判定符合率 | 满足完成判定规范的交付物 / 全部交付物 | 100% |
| 设计-实现追踪闭合率 | 有追踪链的实现项 / 全部实现项 | 100% |
| 仓库实施准入通过率 | 符合实施准入规范的开工项 / 全部开工项 | 100% |

## 5. 评分分层

为避免把“设计完成”误写成“系统完成”，实施项目控制统一使用三类分：

1. 设计基线分
2. 实施准备分
3. 运行准备分
4. 过程控制符合分

当前工作区的 100 分口径只适用于设计基线分。

## 6. 节奏机制

### 6.1 日级节奏

适用于专项补齐与缺口关闭：

1. 交付包进度更新
2. 缺口关闭
3. 文档局部补齐

### 6.2 周级节奏

适用于总控收敛：

1. 总控集成
2. 跨域冲突裁决
3. 复评打分
4. 状态审计

### 6.3 阶段级节奏

适用于冻结定版：

1. 阶段性文档定版
2. 阶段分数确认
3. 阶段状态升级
4. 下一阶段交付清单冻结

## 7. 成本控制规则

### 7.1 阅读控制

1. 一个智能体只读取与本域相关的核心基线。
2. 不允许所有智能体每轮全量读取全部文档。

### 7.2 写入控制

1. 不允许多个智能体同时改同一主文档主段落。
2. 跨域改动必须回到总控统一落地。

### 7.3 上下文控制

1. 优先复用现有基线，不重复建立同类模型。
2. 优先做局部补齐，不进行无必要全局重写。

## 8. 阻塞升级规则

| 问题类型 | 处理方式 |
|---|---|
| 本域缺口 | 专业智能体自行修复 |
| 跨域冲突 | 提交总控裁决 |
| 审计不通过 | 退回对应智能体返工 |
| 状态升级争议 | 由总控和证据验收联合判定 |
| 需人工准入 | 标记 `ready_for_human_acceptance`，等待人工确认 |

## 9. 与 Harness 状态机制的绑定

项目控制中的所有阶段更新，必须绑定 Harness 状态标签，不允许使用模糊描述替代正式状态。

特别约束：

1. `ready_for_human_acceptance` 不是 `complete`
2. `pass_with_runtime_blockers` 不是 `accepted`
3. 任何缺少证据的升级都应退回 `partial` 或 `blocked`

## 10. 本机制的完成判定

本机制完成不代表系统完成。  
其完成标准为：

1. 交付包控制对象完整
2. 量化指标完整
3. 成本、质量、治理三类机制明确
4. 可直接用于后续真实实施和周级控制

自进入真实仓库实施起，项目控制必须额外检查：

1. 是否满足交付物完成判定规范
2. 是否满足仓库实施准入规范
3. 是否建立设计-实现追踪项
4. 是否形成分层测试与验证证据
5. 是否满足状态升级与验收放行规范
