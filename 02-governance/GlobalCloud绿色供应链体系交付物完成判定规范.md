# GlobalCloud 绿色供应链体系交付物完成判定规范

日期：2026-06-08  
状态：交付物完成判定规范 v1  
用途：统一定义各类交付物何时可以被判定为“完成”，防止结构存在即视为完成。

## 1. 适用原则

所有交付物必须同时满足：

1. 有明确目标
2. 有主责人
3. 有输入输出
4. 有证据
5. 有完成判定

缺任一项，不得判定完成。

## 2. 交付物分类

当前交付物统一分为：

1. 文档类
2. 配置类
3. 接口与事件类
4. 数据库与数据类
5. 代码实现类
6. 测试与验证类
7. 验收与审计类

## 3. 五件套判定模板

每个交付物都必须绑定：

1. 交付目标
2. 输入基线
3. 输出结果
4. 证据位置
5. 完成判定条件

## 4. 分类完成标准

### 4.1 文档类

完成条件：

1. 文档范围与目标明确
2. 与当前架构口径一致
3. 有引用或证据来源
4. 有版本日期
5. 已进入文档索引

不算完成的情况：

1. 只有标题或模板
2. 与当前口径冲突
3. 无法落到交付包或验收场景

### 4.2 配置类

完成条件：

1. 配置项定义清楚
2. 环境范围明确
3. 风险项明确
4. 回滚或恢复方式明确
5. 已与对应系统/仓库绑定

### 4.3 接口与事件类

完成条件：

1. 唯一生产者明确
2. 消费者明确
3. 合同字段明确
4. 失败/重试/幂等要求明确
5. 验收方式明确

### 4.4 数据库与数据类

完成条件：

1. 主责系统明确
2. 可写边界明确
3. 禁止事项明确
4. 读模型/主账边界明确
5. 权限与隔离验证要求明确

### 4.5 代码实现类

完成条件：

1. 对应设计条目明确
2. 仓库、分支、文件落点明确
3. 变更边界明确
4. 至少有本地验证或验证计划
5. 未覆盖既有未授权改动

### 4.6 测试与验证类

完成条件：

1. 测试目标明确
2. 输入样本明确
3. 预期结果明确
4. 结果可复核
5. 失败处置明确

### 4.7 验收与审计类

完成条件：

1. 验收对象明确
2. 证据清单完整
3. 结论与状态一致
4. 阻塞项明确
5. 人工确认门清晰

## 5. 与团队执行的绑定

1. 各专项负责人负责提交交付物五件套。
2. 小即负责在控制塔中显示交付物完成度，而不是只显示文档存在度。
3. 评衡与证验负责按本规范驳回“形式存在但未完成”的交付物。

## 6. 强制禁止

不得把以下情况写成完成：

1. 只有文档，没有仓库落点
2. 只有代码，没有设计对应关系
3. 只有接口名，没有合同和验证方式
4. 只有页面，没有状态、证据与验收标准
5. 只有 AI 结论，没有业务或治理证据
