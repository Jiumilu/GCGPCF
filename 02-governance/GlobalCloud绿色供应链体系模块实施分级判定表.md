# GlobalCloud 绿色供应链体系模块实施分级判定表

日期：2026-06-08  
状态：实施分级判定表 v1  
适用范围：P0 最小闭环、`WAES / PVAOS / GPC 当前原型与目标 GPC-Native / GFIS` 四条主线、12 个核心模块。  
使用口径：本表是开发、配置、接入、局部开发、重构的前置门。**没有分级判定，不得进入开发。**

## 1. 判定规则

### 1.1 五级分级

1. **直接可用**  
   现有能力已能直接支撑目标 SOP，不需要改对象模型、状态机、权限边界和体验骨架。
2. **产品内实现**  
   可以通过现有配置、脚本、工作流、模板、参数、低代码能力实现，不改主产品代码。
3. **二次开发**  
   主责系统和产品方向正确，但缺必要对象、接口、状态或视图，需要受控代码增量。
4. **局部开发**  
   现有产品部分可用，但必须通过旁路服务、适配器、中间层或局部增强补足关键缺口。
5. **全面重构**  
   主责错位、对象模型错位、状态机冲突、技术债过大，继续修补的成本高于重构。

### 1.2 判定维度

每个模块必须同时检查：

1. 是否属于 P0 最小闭环必须项
2. 主责系统是否唯一
3. 是否与统一对象模型冲突
4. 是否与统一状态机冲突
5. 是否与统一体验骨架冲突
6. 是否与治理/证据要求冲突
7. 是否可用现有脚本、配置、工作流直接落地
8. 当前继续修补的成本，是否低于局部开发或重构

### 1.3 明确红线

以下情况禁止直接进入开发：

1. 未完成分级判定
2. 未绑定 P0/P1/P2 闭环
3. 未明确主责系统
4. 未明确证据和状态门
5. 用历史原型冒充正式主线

## 3. 第一轮模块实施分级判定总表

| 模块 | P0 必须项 | 主责系统 | 相关系统 | 当前现状 | 分级结论 | 结论理由 | 当前主要风险 | 下一步动作 |
|---|---|---|---|---|---|---|---|---|
| 项目 | 是 | WAES + PVAOS | Brain | WAES 已定义 `SupplyChainProject / ProjectTemplate / ProjectTopology`，PVAOS 已承担项目入口 | **产品内实现** | 项目初始化、模板启用、项目注册、角色绑定主要可由现有对象、配置和治理流承载，不应先改代码 | 项目模板与真实发布验证证据还不足 | 先用现有对象和流程做项目初始化样板 |
| 组织 | 是 | PVAOS | GPC-Native / WAES | PVAOS 已有组织、租户、伙伴、项目入口与服务面 | **直接可用** | 组织对象主责清晰，现有入口与主数据职责成立 | 后续需要统一权限与租户边界 | 直接纳入 P0 闭环，不单独开发 |
| 伙伴 | 是 | PVAOS | GPC-Native | PVAOS 已有伙伴入口，GPC 只消费伙伴关系 | **直接可用** | 伙伴主数据主责清晰，满足生态入口要求 | 与客户/供应商/承运商的角色细分仍需继续梳理 | 直接纳入 P0 闭环，不单独开发 |
| 平台订单 | 是 | 目标 `GPC-Native` | PVAOS / GFIS / WAES | 当前只有 Odoo 原型与协同设计，没有独立 `GPC-Native` 实现仓库 | **全面重构** | 主线目标已切换为 `GPC-Native`，Odoo 原型只能做样本，不能继续当正式主线 | 若继续在 Odoo 原型上修补，会锁死对象模型和体验骨架 | 立项定义 `GPC-Native` 一期平台订单骨架 |
| 工厂订单 | 是 | GFIS | GPC-Native / WAES | GFIS 已承担工厂订单事实，仓库已有真实工厂执行对象证据 | **直接可用** | 主责正确，工厂订单应继续留在 GFIS，不需要迁移 | 与平台订单映射关系仍需清晰化 | 直接进入 P0 闭环，补齐映射与证据 |
| 工单 | 是 | GFIS | WAES | GFIS 仓库已有 `Work Order` 与相关扩展字段和 API 边界 | **直接可用** | 工单主责明确，GFIS 当前方向正确 | 仍缺运行前样本和回执链 | 直接纳入 P0 闭环，补工单证据样本 |
| 质量 | 是 | GFIS | WAES / Brain | GFIS 已有 `Quality Inspection`、隔离、CAPA 设计方向 | **二次开发** | 主责系统正确，但要满足绿链体系的证据、复盘、知识回流，还需补治理与引用能力 | 容易把 AI 草案误当质量事实 | 在 GFIS 上做受控质量闭环增量，不另建质量系统 |
| 库存 | 是 | GFIS | WAES / GPC-Native | GFIS 已有 `Stock Entry`、库存事务、线边库存设计方向 | **直接可用** | 库存主账应继续留在 GFIS，现有方向正确 | 跨系统直写和读模型误写风险高 | 保持 GFIS 主账，外部只做只读投影 |
| 批次 | 是 | GFIS | GPC-Native / WAES | GFIS 已有 `Batch / MaterialLot`、来源追溯和发货关联证据 | **直接可用** | 批次追溯主责清晰，已具备主链基础 | 仍缺跨平台展示一致性与客户侧回指体验 | 直接纳入 P0，后续补追溯视图一致性 |
| 发货 | 是 | GFIS | GPC-Native / WAES | GFIS 负责 `FactoryShipmentRelease`，GPC 负责运输协同 | **产品内实现** | 业务主责清楚，主要是对象映射、回执、交接、证据链配置问题，不应先重写主逻辑 | 工厂发货与平台运输衔接不清会形成双主账错觉 | 用现有对象、事件、证据链先打通交接 |
| 签收 | 是 | GPC-Native | GFIS / WAES | 当前只有 Odoo 原型与协同设计，目标主线尚未落为 `GPC-Native` | **全面重构** | 签收是链侧协同事实，应归 `GPC-Native`，不能继续依赖原型拼凑 | 若沿 Odoo 原型走，会把签收、POD、争议处理继续绑死 | 在 `GPC-Native` 一期中单独落签收与 POD 主线 |
| 异常 | 是 | 分域主责：GFIS / GPC-Native；治理归 WAES | Brain / Hermes / XGD | 当前异常对象已定义，但运行处理流与跨域关闭口径仍分散 | **局部开发** | 异常不能全面重构，也不能只靠配置，需要围绕主责系统加统一异常壳、证据壳、复盘壳 | 最容易出现多系统各关一半的拼凑感 | 建统一异常关闭契约和异常工作台，不重写各主责系统内核 |

## 4. 分级结论汇总

### 4.1 直接可用

1. 组织
2. 伙伴
3. 工厂订单
4. 工单
5. 库存
6. 批次

特点：

- 主责系统已清晰
- 现有对象方向正确
- 不需要先改代码
- 先纳入 P0 闭环即可

### 4.2 产品内实现

1. 项目
2. 发货

特点：

- 应优先通过现有对象、模板、配置、事件与证据链落地
- 不应一开始进入代码重构

### 4.3 二次开发

1. 质量

特点：

- 主责系统正确
- 但要满足治理、证据、知识回流和统一体验骨架，现有能力还不够

### 4.4 局部开发

1. 异常

特点：

- 不能重写所有异常逻辑
- 也不能只靠现有页面拼装
- 需要做统一异常壳层

### 4.5 全面重构

1. 平台订单
2. 签收

特点：

- 当前正式主线应是 `GPC-Native`
- Odoo 原型只能作为样本，不适合作为继续实施主线
- 这是整个体系最重要的重构入口

4. 明确项目初始化与发货交接先走产品内实现，不先改代码。
