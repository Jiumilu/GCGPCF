# GlobalCloud 绿色供应链体系多智能体实施团队与协同方案

日期：2026-06-07  
状态：实施组织设计 v1  
适用范围：GlobalCloud 绿色供应链体系总体实施、专项设计补齐、基线收口、联调准备、试运行前治理控制。  
治理口径：遵循 Harness Engineering 项目治理机制，由总负责人 **小即** 统一分派、集成、复评和状态升级。

## 1. 方案目标

本方案用于把 Codex 的执行能力组织成可治理的实施团队，目标是：

1. 提升整体推进速度，但不牺牲边界清晰度。
2. 将架构、对象、事件、数据库、知识、AI、验收和证据分域并行推进。
3. 用总控和审计机制避免多智能体交叉改写、重复生产和状态漂移。
4. 将设计推进、质量门禁、量化评分、项目控制和 Harness 状态机制统一到一套方法中。

## 2. 总体原则

### 2.1 先治理，再并行

并行前必须先冻结：

1. 体系边界
2. 交付包边界
3. 文档边界
4. 状态标签
5. 验收口径
6. 证据口径

### 2.2 一智能体一目标

每个智能体只负责单一清晰目标，必须明确：

1. 输入
2. 输出
3. 负责文档或对象域
4. 禁止动作
5. 质量门禁

### 2.3 总控统一集成

并行智能体可以补齐和分析，但以下动作只能由总控执行：

1. 统一命名
2. 跨域裁决
3. 总体评分
4. 状态升级
5. 定版发布

### 2.4 结构通过不等于运行完成

设计基线满分只表示设计完备，不表示系统已联调、已试运行或已规模化运行。

## 4. 协同方式

### 4.1 星型协同

采用总控居中、专业智能体围绕总控的星型协同模式：

1. 专业智能体只对总控提交结果。
2. 不鼓励专业智能体互相直接改写对方主文档。
3. 审计智能体独立复核，不参与主设计产出。

### 4.2 协同轮次

#### 第一轮：基线冻结

总控分发当前基线和边界，专业智能体只识别本域缺口。

#### 第二轮：专项补齐

各智能体补齐：

1. 对象
2. 事件
3. 数据边界
4. 验收场景
5. 风险清单

#### 第三轮：总控集成

总控统一：

1. 命名
2. 主责
3. 状态标签
4. 文档索引
5. 跨域裁决

#### 第四轮：审计复核

评估审计和证据验收分别进行评分和阻塞项检查。

#### 第五轮：冻结定版

总控输出阶段性定版文档集、评分结果和下一阶段执行清单。

## 5. Harness Engineering 映射

每个实施交付包必须至少具备以下治理资产：

1. `PROJECT_HARNESS_MANIFEST.md`
2. `docs/harness/acceptance.md`
3. `docs/harness/evidence/README.md`
4. `docs/harness/status-audit-YYYY-MM-DD.md`

### 5.1 统一状态标签

全体智能体统一使用：

- `not_started`
- `in_progress`
- `partial`
- `blocked`
- `pass_with_runtime_blockers`
- `ready_for_human_acceptance`
- `accepted`
- `complete`

### 5.2 唯一状态升级者

只有小即在审计智能体确认后，才能升级总体状态。

## 6. 写入边界

### 6.1 专业智能体允许写入

1. 本域专项设计文档
2. 本域对象、事件、验收补充段
3. 本域风险和建议

### 6.2 专业智能体禁止写入

1. 擅自修改跨域主责边界
2. 擅自将总体状态升级为 `complete`
3. 擅自改变 WAES/GPC-Native/GFIS/PVAOS/Edge 的核心定位
4. 将 AI 建议写成业务事实

## 8. 当前阶段适用策略

对于当前工作区，建议采用：

1. 设计基线阶段：11 角色完整方案
2. 实施准备阶段：按交付包收缩为 6 角色执行单元
3. 真实联调阶段：总控 + 交付包牵头 + 审计双角色最小闭环

## 9. 本方案的验收结论

本方案建立的是多智能体实施组织和协同机制，不是实际运行完成结论。  
其完成判定标准为：

1. 角色分工明确
2. 写入边界明确
3. 状态机制明确
4. 审计机制明确
5. 与 Harness Engineering 治理口径一致
