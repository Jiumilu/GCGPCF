# GlobalCloud 绿色供应链平台主架构与宪法约束融合建议

日期：2026-06-07
状态：架构主线更新稿 v1
适用范围：当前工作区全部绿色供应链体系设计文档。

## 1. 文档定位

本文不是替代现有体系主方案，而是给当前工作区内所有文档提供统一的架构主线：

1. 以现有绿色供应链平台架构为主。
2. 以宪法和治理规则作为上位约束。
3. 不把宪法写成业务平台。
4. 不把平台写成治理中心。

## 2. 当前统一主口径

### 2.1 三层主架构

1. 治理与监控层：
   - 主责系统：`WAES`
   - 主责内容：规则、治理、证据、状态、指标、AI 授权、发布验证、连接器治理
2. 运营与协同层：
   - 主责系统：`GPC`
   - 生态入口：`PVAOS`
   - 主责内容：平台订单、样品申请、客户签样、转量产、ASN、预约、运输、POD、外部异常、跨组织协同
3. 生产与执行层：
   - 主责系统：`GFIS`
   - 边缘接入：`Edge`
   - 主责内容：配方研发、样品打样、样品检测、工厂订单、工单、质量、库存、批次、LES、EAM、发货出库、现场执行

### 2.2 平台主线定位

当前“绿链平台”应以 `GPC` 为主线，不应回退到“大一统平台”思路。

建议明确：

1. `GPC` 是绿色供应链平台的一期主线平台。
2. `PVAOS` 是入口，不是主业务平台。
3. `WAES` 是治理平面，不是订单和事务平台。
4. `GFIS` 是工厂事实平台，不是区域协同平台。
5. `Edge` 是接入层，不是业务平台。
6. KDS 为企业级知识主存（canonical source），Brain 为知识 UI 层；LLM Wiki 为知识编制工具，不直接等于企业级知识真源。

## 3. 宪法内容应如何进入当前架构

宪法内容不应直接覆盖业务架构，而应转译为 6 类硬约束：

1. 证据分级约束：
   - `E0/E1/E2` 只能作线索
   - 强经营结论必须绑定 `source_refs` 和更高证据等级
2. 主责系统唯一约束：
   - 一个业务事实只能有一个主责系统
3. 治理审批与业务审批分离约束：
   - WAES 不承办工单、质量、库存、发货、签收等事务审批
4. 状态升级约束：
   - `accepted`、`complete` 必须有治理门和证据门
5. 连接器治理约束：
   - 未登记连接器不得进入默认运行态
6. AI 授权约束：
   - AI 建议不能直接写业务事实

## 4. 对绿链平台架构的具体分析建议

### 4.1 平台边界建议

`GPC` 只做 6 类核心能力：

1. 平台订单接入与分发。
2. ASN、预约、车辆、承运商协同。
3. 厂外运输、在途、POD、回单。
4. 外部异常和跨组织例外闭环。
5. 多厂分单、库存可见性、产能协同。
6. 与 GFIS、PVAOS、WAES 的连接器和事件协同。

`GPC` 不做：

1. 工单释放。
2. 质量放行。
3. 库存主账。
4. 设备控制。
5. 证据确证。
6. 治理审批。

### 4.2 WAES 的正确写法

`WAES` 应只写成以下角色：

1. 项目发起与模板启用中枢。
2. 发布验证与阶段验收中枢。
3. 规则、指标、状态、证据和 AI 授权中枢。
4. 控制塔与治理视图中枢。

`WAES` 不应写成：

1. 业务事务审批中心。
2. 订单执行系统。
3. 工厂执行系统。
4. 运输签收系统。

### 4.3 GFIS 的正确写法

`GFIS` 应继续作为单厂本地执行事实源。

建议继续坚持：

1. 配方研发、样品打样、样品检测、工厂订单、工单、质量、库存、批次、LES、EAM、发货出库由 GFIS 负责。
2. GFIS 对外发布事实事件，不接受平台越权改写。
3. GFIS 与 GPC 通过事件和连接器闭环，不共享业务数据库。
4. Edge 信号必须先进入 GFIS 规则，再决定是否形成业务动作。

### 4.4 PVAOS 的正确写法

`PVAOS` 应固定为生态入口层，不应再被写成综合业务平台。

主责限定为：

1. 租户、组织、伙伴、项目、门户接入。
2. 角色授权和入口身份上下文。
3. 对外门户、合作方访问入口。

### 4.5 Edge 的正确写法

`Edge` 是边缘接入层，不是业务账本层。

主责限定为：

1. 采集。
2. 协议转换。
3. 缓存。
4. 补传。
5. 去重。
6. 回执。

## 5. 应进入各专项文档的统一架构更新点

### 5.1 对象目录

应强化：

1. `GPC` 作为平台主线系统。
2. `WAES` 作为治理对象主线系统。
3. `BusinessApprovalReference` 与 `GovernanceApproval` 的分离。
4. 平台对象与工厂对象的映射字段，而非同库共享。

### 5.2 事件合同

应强化：

1. 平台事件只表达平台事实。
2. 工厂事件只表达工厂事实。
3. 治理事件只表达治理事实。
4. AI 事件只表达建议和授权状态。

### 5.3 连接器合同

应强化：

1. `PVAOS -> GPC`
2. `GPC -> GFIS`
3. `GFIS -> GPC`
4. `GFIS/GPC -> WAES`
5. `Edge -> GFIS`
6. `Brain/XiaoC（蚁后）/Hermes/XGD（大象） -> WAES`

### 5.4 SOP 模板

应强化：

1. 每个 SOP 必须显式填写主责系统。
2. 必须分开业务确认点与治理确认点。
3. AI 允许动作与禁止动作必须显式声明。

### 5.5 验收矩阵

应强化：

1. 平台完成不等于业务完成。
2. 配置完成不等于运行完成。
3. 证据齐全不等于业务主账正确，仍需主责系统回执。

### 5.6 数据层与知识层

应强化：

1. 数据层升级为业务事实层、资源仓库层、企业级知识主存层、知识引擎层和技术运行底座，其中资源仓库当前已定义十一池，后续可继续新增其它池。
2. `LLM Wiki` 与 `Brain` 当前并行测试，但都不直接承担企业级知识真源角色。
3. `WAES` 对知识准入、发布、引用和 AI 可见范围负责治理。

## 6. 当前最需要你重点看的架构变化

如果只看更新后的架构，建议重点看下面 4 点：

1. `GPC` 被正式收敛为绿色供应链平台主线，而不是历史原型的延伸。
2. `WAES` 被正式收敛为治理平面，而不是业务平台。
3. `GFIS` 与 `GPC` 的“平台事实 / 工厂事实”分层被继续拉开。
4. 宪法内容被下沉为门禁、状态、证据、授权和连接器治理约束，而不是抽象口号。
5. 数据层进一步升级为“企业级知识主存层 + 知识引擎层”的双层知识架构。

## 7. 结论

当前最合理的总设计不是“再造一个更大的平台”，而是：

1. 让 `GPC` 成为绿链平台业务主线。
2. 让 `WAES` 成为治理和控制塔主线。
3. 让 `GFIS` 成为工厂执行主线。
4. 让宪法成为这些系统之间的上位约束，而不是其中任何一个系统的替代物。
